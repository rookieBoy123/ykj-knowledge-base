# 订单退款完成与跨渠道退款分派流程

> 文档版本 2026-07-05 | 面向读者：产品经理、后端开发、运营

## 概述

当订单发生退款（用户取消/超时取消/重复支付自动退款/管理后台审核退款）时，系统通过事件驱动的方式完成从"退款申请创建"到"退款完成"的完整流程。本流程覆盖了多支付渠道（微信/支付宝）的退款执行、同一订单多笔流水的独立退款、以及退款回调后的状态同步。

## 触发条件

以下场景会触发退款流程：

- **用户主动取消订单**：订单已支付状态下用户取消
- **订单超时取消**：支付倒计时归零，系统自动取消
- **重复支付自动退款**：用户对同一订单多次支付，系统自动为多余支付创建退款
- **管理后台审核退款**：管理员在后台审核通过退款申请

## 步骤详解

### 第一步：退款申请单创建

系统通过 `OrderRefundApplyCreateEvent` 事件触发退款申请单的创建。事件中携带以下关键信息：

| 字段 | 说明 |
|------|------|
| 订单编号 | 关联的订单 |
| 支付流水号（wxTransactionId） | 标识是哪一笔支付需要退款 |
| 退款金额（refundAmount） | 退回多少钱 |
| 支付会员ID | 钱退给谁 |

**唯一性保护**：同一笔支付流水（订单号 + 支付流水号）只能创建一条退款申请单。重复提交会提示"该订单已提交过退款信息"。但同一订单的不同支付流水可以各自创建独立的退款申请单。

**代码来源**：`RefundApplyServiceImpl.createRefundApply()` 第 75-99 行

### 第二步：退款审核（管理后台触发）

管理员在退款审核页面有两种处理方式：

**同意退款**：
- **手动退款**（Manual_Refund）：管理员线下退款（如银行转账），审核通过后直接标记"已退款"，不调用第三方接口
- **原路退回**（Bracktrack）：系统自动调用支付宝或微信的退款接口，将钱原路退回用户账户

**拒绝退款**：
- 必须填写审核备注（系统强制校验）
- 申请单状态改为"审核不通过"
- 发布 `RefundApplyApprovedEvent`（agree=false）通知相关业务模块恢复订单状态

**代码来源**：`RefundApplyServiceImpl.agreeRefundApply()` 第 106-139 行；`rejectRefundApply()` 第 145-167 行

### 第三步：按支付渠道分派退款

批准退款后，系统根据支付渠道执行不同的退款逻辑：

```
                 ┌──────────────────┐
                 │  退款申请单审核通过  │
                 └────────┬─────────┘
                          │
                 ┌────────▼─────────┐
                 │  resolvePayChannel │
                 │  解析支付渠道       │
                 └───┬──────────┬────┘
                     │          │
           ┌────────▼──┐   ┌───▼─────────┐
           │ 微信退款    │   │ 支付宝退款    │
           │ (wxRefund  │   │ (alipay     │
           │  Service)  │   │  Service)   │
           └────────┬───┘   └───┬─────────┘
                     │          │
                     ▼          ▼
              调用微信/支付宝退款接口
                     │
                     ▼
              退款状态改为"退款中"
                 (REFUNDING)
```

**代码来源**：`RefundApplyServiceImpl.doRefund()` → `executeRefundByChannel()` 第 360-374 行

### 第四步：退款回调处理

退款接口调用成功后，支付宝/微信会异步回调通知系统退款结果。系统通过事件监听器处理：

1. **`RefundCompletedEvent` 到达**：由 `handleRefundedEvent` 方法监听
2. **更新退款申请单**：状态改为"已退款"，记录退款时间
3. **发布 `RefundApplySucceedRefundedEvent`**：通知订单模块更新订单状态

**支付渠道优先级**：回调事件中会携带 `payChannel` 参数。如果事件携带了支付渠道（新逻辑），直接使用；如果未携带（兼容旧事件），回退查询数据库获取支付渠道。

**代码来源**：`RefundApplyServiceImpl.handleRefundedEvent()` 第 262-290 行

### 第五步：订单状态更新

收到退款成功事件后，订单模块执行以下操作：

1. 使用 `LambdaUpdateWrapper`（非乐观锁）更新订单状态为"已退款"（REFUNDED）
2. 设置退款时间
3. 如果更新失败（影响行数为0），记录错误日志并抛出异常
4. 发布 `OrderRefundEvent` 通知其他业务模块（如问诊状态更新、会员权益调整等）

**代码来源**：`OrderCommandServiceImpl.completeRefund()` 第 631-653 行

## 异常处理

| 情况 | 系统处理方式 |
|------|------------|
| 退款申请单状态不允许审核 | 后端拦截，提示"申请单不存在或状态不允许审核" |
| 拒绝退款未填备注 | 后端拦截，提示"请填写备注" |
| 非原路退回类型申请重试 | 后端拦截，提示"当前退款申请单非原路退回申请单" |
| 退款接口调用失败 | 退款状态改为"退款失败"，记录失败原因，不中断流程 |
| 退款回调更新申请单失败（乐观锁） | 抛出异常"退款申请信息已变更，请刷新后重试" |
| 同一支付流水重复创建退款申请 | 后端拦截，提示"该订单已提交过退款信息" |
| 重复支付退款时某笔流水已处理过 | 跳过该流水，不重复退款 |

## 关联功能

- **订单取消流程**：取消订单时触发退款申请创建
- **重复支付处理**：`completePayment` → `doCompletePayment` 中自动退款
- **退款审核管理**：管理后台退款审核页面调用 `agreeRefundApply` / `rejectRefundApply`
- **退款重试**：`retryRefund` 方法支持对失败的退款进行重试
- **手动退款转标记**：`manualRefund` 方法将失败的原路退回转为手动退款标记

## 退款申请单状态流转

```
创建 → 待审核(BE_AUDIT)
         ├──→ 审核通过(AUDIT_PASS) → 退款中(REFUNDING) → 已退款(REFUNDED)
         │                        ├──→ 手动退款 → 已退款(REFUNDED)
         │                        └──→ 退款失败(REFUND_FAIL) → 重试 → 退款中 → 已退款
         │                                                     └──→ 手动退款标记
         └──→ 审核不通过(AUDIT_FAIL) → 订单状态恢复为退款前状态
```
