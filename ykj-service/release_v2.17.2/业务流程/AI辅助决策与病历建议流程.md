# AI辅助决策与病历建议流程（服务端）

> 文档版本：v1.0 | 面向读者：产品/运营

## 概述

平台为问诊中的医生提供两项AI辅助功能：**提问建议**（智能推荐问诊中可向患者提出的问题）和**病历辅助**（根据问诊对话自动生成病历草稿）。两个功能均受独立开关控制，且都有失败次数限制和锁定机制，防止AI服务异常时频繁重试消耗资源。

## 触发条件

### 前置条件
- 系统开启了对应功能的开关（`questionAiSuggestion` / `medicalAiSuggestion`）
- 存在有效的问诊单（问诊中的状态下医生可使用）
- 该问诊单在过去30分钟内的AI调用失败次数未超过3次

### 模块归属
`ykj-service-inquiry` → `SuggestionServiceImpl`

## 流程详解

### AI提问建议（getQuestions）

**第一步：开关检查**

系统首先检查配置开关 `questionAiSuggestion`。如果未开启，直接返回错误提示「系统暂不支持医生问答辅助，请联系管理员」。这是一个硬拦截 —— 管理员未配置该功能时，医生端不会出现AI提问建议入口。

**第二步：锁定状态检查**

查询Redis缓存中是否存在该问诊单的「锁定」标记：
- 如果已锁定：返回失败结果 + 锁定时间戳（lockTime），前端据此展示倒计时
- 锁定条件：30分钟内AI调用失败次数达到3次

**第三步：失败次数检查**

查询Redis中该问诊单的累计失败次数：
- 次数 ≥ 3：自动写入锁定标记（30分钟有效），返回锁定结果
- 次数 < 3：继续向下执行

**第四步：构建请求并调用AI**

系统从多个数据源组装请求参数：
1. **患者信息**：从问诊单获取患者ID → 查询患者详情 → 从身份证提取年龄和性别
2. **病史信息**：三层数据源优先级合并：
   - 患者基础病史（过敏史/现病史/个人史）
   - AI预问诊信息（仅限状态为「已生成报告」的预问诊）
   - 已有病历信息（从病历表补充）
3. **聊天记录**：从IM群组获取历史消息，过滤出医生和患者的文本消息，按时间排序
4. **病情描述**：优先取病历中的病情描述，无则取问诊单的描述

**第五步：处理AI响应**

- 成功（返回码200 + 数据非空）：删除Redis中的失败次数记录，返回最多9条建议问题
- 失败：记录错误日志，累计失败次数+1（首次设为1），返回失败结果

### AI病历辅助（getMedicalRecord）

流程与提问建议基本相同，差异在于：

1. **不同的开关**：使用`medicalAiSuggestion`配置控制
2. **不同的AI接口**：调用病历辅助接口（`suggesterService.medicalRecord`）
3. **不同的缓存key**：使用`medical-record`类型后缀区分于提问建议的`question`
4. **额外的后处理**：AI返回的诊断名称需要与标准诊断库做匹配对齐，通过`diagnosisService.findByNamesAndType`查找对应的标准诊断编码

### 锁定与重试机制

两个功能共享相同的锁定策略，但独立计数：

| 规则项 | 说明 |
|--------|------|
| 失败计数窗口 | 30分钟（ERROR_LOCKING_TIME） |
| 锁定触发阈值 | 累计失败3次 |
| 锁定持续时间 | 30分钟 |
| 成功重置 | 任意一次成功调用立即清空失败计数 |
| 两个功能是否互斥 | 否，提问建议和病历辅助各自独立计数和锁定 |

锁定的设计意图：AI服务底层可能出现不可用或持续异常，频繁重试会加重系统负担且无意义，锁定机制强制冷却一段时间后再允许重试。

## 数据来源优先级

病史信息组装采用`StringUtils.defaultIfBlank`策略（前者不为空则使用前者）：

1. 第一优先级：患者基础信息中的病史（基础信息表）
2. 第二优先级：AI预问诊生成的病史（仅状态=已生成报告时使用）
3. 第三优先级：已有病历中的病史（病历表）

三层合并确保病史信息尽可能完整，但不会互相覆盖。

## 异常处理

| 情况 | 系统处理方式 |
|------|-------------|
| 功能开关未开启 | 抛出业务异常，前端展示错误提示 |
| 已达3次失败/锁定中 | 返回锁定时间戳，前端展示倒计时 |
| AI服务调用超时/异常 | 捕获异常不抛出，记录日志+累计失败次数 |
| AI返回空数据 | 累计失败次数（非成功不算重置） |
| 问诊单不存在 | 抛出「没有相关问诊单」异常 |
| 患者信息不存在 | 抛出「没有患者信息」异常 |
| IM消息获取失败 | 捕获异常，使用默认空消息（仅含病情描述） |
| 预问诊/病历信息获取失败 | 捕获异常，跳过该数据源（不影响主流程） |

## 设计意图

- **静默降级**：所有子数据源（预问诊/病历/IM消息）的获取失败都不影响主流程，确保AI建议始终有兜底数据
- **失败保护**：3次失败30分钟锁定，防止无效重试打满QPS
- **诊断对齐**：病历辅助中AI返回的诊断名称会与标准诊断库匹配，确保诊断编码的规范性
