# 退款申请审核与处理流程

> 文档版本 2026-07-02 | 面向读者：运营人员、财务人员

## 概述

当患者需要退款时（如药品缺货、问诊取消、支付重复等场景），系统生成退款申请单，由管理员审核后执行退款。退款支持两种方式：原路退回（通过微信支付/支付宝自动退款到患者账户）和手动退款（管理员线下退款后标记）。

## 触发条件

- **触发场景**：
  - 患者取消订单后自动触发退款（如果有自动退款配置）
  - 管理员在后台手动发起退款
  - 重复支付场景（系统检测到同一订单被多次支付，自动退款多余款项）
- **前置条件**：订单必须已支付

## 步骤详解

### 第一步：创建退款申请

退款申请单包含以下关键信息：
- 订单编号（同一订单允许有多个退款申请，但同一笔支付流水只允许一条）
- 退款金额
- 退款原因备注
- 微信支付流水号（用于原路退回）
- 支付渠道信息

系统生成唯一的退款申请单号（格式：Refund+日期+随机数），初始状态为"待审核"。

### 第二步：审核环节（管理员操作）

管理员对退款申请做出两种决策：

**同意退款**：
- 必须填写审核备注
- 选择退款方式：
  - **原路退回**：系统自动调用微信/支付宝退款接口，钱款原路返回患者支付账户
  - **手动退款**：管理员线下处理退款后在此标记，系统不调用支付接口
- 审核通过后更新状态并发布事件通知相关模块

**拒绝退款**：
- 必须填写拒绝备注（说明原因）
- 状态变更为"审核拒绝"
- 发布拒绝事件通知

### 第三步：执行原路退款

系统根据支付渠道自动选择退款路径：
- **微信支付订单**：调用微信退款接口，传入支付流水号和退款金额
- **支付宝订单**：调用支付宝退款接口

退款执行前校验：
- 退款申请必须已审核通过
- 退款状态必须是"未退款"或"退款失败"（允许重试）
- 退款方式必须是原路退回

退款金额以元为单位（系统自动处理分→元的转换）。

### 第四步：退款结果处理

**退款成功**：
- 退款申请状态更新为"已退款"，记录退款时间
- 发布退款成功事件，通知订单模块做后续处理（如取消订单、恢复库存等）

**退款失败**：
- 退款申请状态更新为"退款失败"，记录失败原因
- 管理员可以重新发起退款（重试机制）
- 也可以改为手动退款方式处理

## 状态流转

```
待审核 → 审核通过 → 退款中 → 已退款
                 ↘ 审核拒绝
                 
审核通过 + 退款失败 → 可重试（回到"退款中"）
审核通过 + 退款失败 → 可转手动退款（标记为"已退款"）
```

## 异常处理

| 情况 | 系统处理方式 |
|------|------------|
| 同一订单+支付流水重复提交退款 | 提示"该订单已提交过退款信息" |
| 退款申请状态不允许审核 | 提示"申请单不存在或状态不允许审核，请联系管理员" |
| 拒绝退款时未填写备注 | 提示"请填写备注" |
| 重试退款时未找到退款申请 | 提示"未找到退款申请单, 无法重试退款" |
| 手动退款时状态异常 | 提示"未找到退款申请单或申请单状态异常, 无法退款" |
| 退款申请非原路退回类型 | 提示"当前退款申请单非原路退回申请单, 无法进行原路退回操作" |
| 原路退款第三方接口异常 | 自动记录失败原因，退款申请状态更新为"退款失败"，需管理员介入 |
| 退款申请信息并发更新冲突 | 提示"退款申请信息已变更，请刷新后重试" |
| 自动退款异常 | 日志记录详细错误信息，提醒管理员关注 |

## 关联功能

- **订单管理**：退款结果会联动更新订单状态
- **交易明细查询**：退款流水会在财务看板中展示
- **支付模块**：微信支付和支付宝的退款接口对接
- **库存管理**：退款成功后可能触发库存恢复
