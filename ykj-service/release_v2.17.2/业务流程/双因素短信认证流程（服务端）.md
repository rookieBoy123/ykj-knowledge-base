# 双因素短信认证流程（服务端）

> 文档版本 2026-07-03 | 面向产品、运维

## 概述

双因素认证（TFA）是管理员登录后台系统时的额外安全保障。当管理员在高风险场景下登录（如异地登录、新设备登录），系统除了要求输入账号密码外，还会要求输入发送到管理员绑定手机号的短信验证码，确保登录人是账号的真实持有者。

## 触发条件

- **触发时机**：管理员在登录流程中被系统判定为需要二次验证时
- **服务模块**：`ykj-service-system`
- **接口前缀**：`/mgmt/tfa/sms`

## 步骤详解

### 第一步：发送短信验证码

管理员在后台登录页面触发双因素认证后，前端调用发送验证码接口：

- **操作**：系统向当前管理员绑定的手机号发送一条短信验证码
- **返回**：系统返回脱敏后的手机号（如 138****1234），方便管理员确认短信发送到了哪个号码
- **安全设计**：验证码只能发给当前已登录的管理员绑定的手机，不能由前端指定手机号——防止验证码被恶意转发

### 第二步：输入验证码

管理员查看手机短信，在页面中输入收到的验证码。

### 第三步：验证验证码

前端将验证码提交到验证接口：

- **验证要素**：管理员的登录Token（从请求头中提取） + 输入的短信验证码
- **验证流程**：系统比对验证码是否正确、是否在有效期内（通常60秒或5分钟）、Token是否有效
- **返回**：验证通过返回 `true`，验证失败返回 `false`

### 第四步：完成登录

验证通过后，管理员正常进入后台系统。如果验证失败，管理员可以重新获取验证码重试（通常有次数限制）。

## 安全机制

| 机制 | 说明 |
|---|---|
| 手机号绑定 | 验证码只能发送到管理员已绑定的手机号，无法指定其他号码 |
| Token绑定 | 验证时关联管理员的登录Token，防止验证码被截获后在其他设备上使用 |
| 脱敏返回 | 发送验证码后返回脱敏手机号，即使接口被截获也不会暴露完整手机号 |
| 无权限控制 | 这两个接口（发送/验证）在登录流程中调用，不需要额外权限——因为它们本身就有Token校验 |

## 异常处理

| 情况 | 系统处理方式 |
|---|---|
| 管理员未绑定手机号 | 发送验证码失败，需先在个人信息中绑定手机号 |
| 验证码输入错误 | 返回 `false`，前端提示"验证码错误" |
| 验证码过期 | 返回 `false`，需重新发送验证码 |
| 重复发送过频 | 由短信服务层面的频率限制控制（通常60秒内只能发送一次） |
| 登录Token失效 | 验证失败，需重新走登录流程 |

## 关联功能

- **管理员登录流程**：双因素认证是登录流程的安全增强环节
- **短信服务**：依赖第三方短信网关发送验证码
- **用户管理**：管理员的手机号在用户管理中绑定和维护
