# 药房管理与药品校验流程（小程序端）

> 文档版本 v1.0 | 面向读者：产品运营

## 概述

医生在开处方时需要选择药房，系统根据医生的药房授权、药品库存、加工服务等因素，智能排序药房列表，帮助医生快速选择合适的药房。患者端也可查询药房信息。

## 触发条件

- 医生开方时选择药房
- 患者查看药房信息

## 流程详解

### 一、医生查询授权药房

医生端查询药房时，系统仅返回该医生被授权的药房列表：
1. 系统根据当前登录医生的ID查询已绑定的药房
2. 如果医生没有任何药房授权，返回空列表
3. 查询到的药房默认过滤条件为"启用代煎服务"（decoctServiceEnabled=true）

**权限控制**：只有医生角色（ROLE_DOCTOR）可以查询药房是否绑定、查询授权药品类型。

### 二、按药品信息智能推荐药房（listByDrugs）

这是最核心的药房推荐接口，用于医生添加药品后选择合适的药房：

1. **传入参数**：药品类型（成药/中药）、药品清单（JSON数组，含监管码和药品别名）
2. **查询药房**：先获取医生授权的所有药房列表
3. **组装加工服务**：为每个药房附加可用的加工服务信息（加工工艺名称、加工天数、单价等）
4. **校验药品**：逐个检查药品清单在每家药房的情况：
   - **药品齐全**：该药房有此药品且库存充足
   - **库存不足**：该药房有此药品但当前库存少于需求数量（且该药房启用了库存管理）
   - **无此药品**：该药房完全没有此药品
5. **智能排序**（评分从高到低）：
   - 药品齐全 +4分
   - 有药品但库存不足 +2分
   - 无此药品 +0分
   - 有加工服务 +1分

排序结果：优先推荐药品齐全的药房，其次是部分缺货的药房，最后是没有该药品的药房。

### 三、药房绑定校验

医生可以选择药房后，通过 `_checkBinding` 接口验证该药房是否仍在其授权范围内。如果药房已被管理员解除授权，返回false。

### 四、查询授权药品类型

通过 `enableDrugsType` 接口，医生可以查询某家药房支持哪些药品类型（如成药、中药等），用于开方时判断该药房是否支持当前处方类型。

### 五、查询药房加工服务

通过 `{id}/services` 接口查询指定药房的代煎加工服务列表，包括加工类别、加工工艺、计费方式和价格信息。

### 六、计算加工费

通过 `computeProcessingServiceFee` 接口，传入药房ID、加工工艺ID和计算值（如药品重量），系统计算出该药房的加工服务费用。

计费方式有两种：
- **按剂收费**：加工约N天，X元/剂
- **按重量收费**：加工约N天，X元/每克（起算重量）

### 七、患者端药房查询

患者端查询药房时，如果当前角色是患者（非医生），则返回**所有**药房列表，不限制授权范围。

## 异常处理

| 情况 | 系统处理方式 |
|------|-------------|
| 医生无任何药房授权 | 返回空列表 |
| 药房启用了库存管理但库存不足 | 药品归入"库存不足"分类，仍显示该药房但排序靠后 |
| 药房未启用库存管理 | 即使库存数字不足也不会归入"库存不足"，正常显示 |
| 药品清单为空 | 跳过药品校验，仅返回药房基本信息和加工服务 |
| 医生角色查询未绑定的药房 | checkBinding返回false |

## 关联功能

- 处方开方 → 选择药房后进入下一步开具处方
- 管理后台药房管理 → 管理员配置药房基础信息和授权
- 药品库存管理 → 库存数据影响药房推荐排序
- 加工服务管理 → 加工工艺和类别配置
