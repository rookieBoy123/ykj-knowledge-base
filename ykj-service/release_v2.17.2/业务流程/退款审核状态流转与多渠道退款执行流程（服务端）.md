# 退款审核状态流转与多渠道退款执行流程（服务端）

> 文档版本 2026-07-05 | 面向后端开发者/运维人员

## 概述

退款申请从创建到最终退款完成，经历「审核状态」和「退款状态」两条独立的状态流转线。系统支持两种退款方式：原路退回（自动调用微信/支付宝退款接口）和手动退还（线下处理，系统仅记录状态）。同一笔支付流水只允许创建一条退款申请，防止重复退款。

## 触发条件

- 用户在小程序或管理后台发起退款
- 系统通过事件驱动自动创建退款申请（`OrderRefundApplyCreateEvent`）
- 支持自动退款模式（事件携带 `autoRefund=true` 时，创建后自动审核通过并执行退款）

## 双状态流转机制

退款申请有两个独立的状态维度：

### 审核状态（flow_status）

```
待审核(10) ──审核通过──→ 审核通过(20)
    │                        │
    └──审核不通过──→ 审核不通过(30)
```

### 退款状态（refund_status）

```
未退款(10) ──开始退款──→ 退款中(20) ──退款成功──→ 已退款(30)
    │                        │
    │                        └──退款失败──→ 退款失败(40)
    │                                            │
    │                               （手动退款）   │
    └──────────────────────────────────────────────┘
```

注意：手动退还（Manual_Refund）直接从"未退款"跳到"已退款"，不经过"退款中"状态，因为线下操作不产生系统退款流水。

### 两种退款方式的区别

| 特性 | 原路退回（Bracktrack） | 手动退还（Manual_Refund） |
|------|----------------------|--------------------------|
| 退款渠道 | 微信支付/支付宝自动退款 | 线下处理（如银行转账） |
| 退款状态 | 未退款 → 退款中 → 已退款 | 未退款 → 已退款（跳过退款中） |
| 系统流水 | 有退款流水记录 | 无退款流水 |
| 审核时设置 | 审核通过后自动发起退款调用 | 审核通过即标记已退款 |
| 退款失败处理 | 可重试（retryRefund） | 不适用（不走系统退款） |
| 切换到手动退款 | — | 原路退回失败后可手动退款（manualRefund） |

## 步骤详解

### 第一步：创建退款申请

通过事件驱动或直接调用 `createRefundApply` 创建退款申请：

**去重校验**：
- 同一订单号（`orderNo`）+ 同一自编号（`customNo`）+ 同一微信交易号（`wxTransactionId`）只允许一条记录
- 如果已存在 → 提示"该订单已提交过退款信息"

**初始状态**：
- `flow_status = 10`（待审核）
- `refund_status = 10`（未退款）
- `refundOrderNo`：自动生成（格式 `Refund{yyyyMMdd}{7位随机数}`，冲突时递归重试）

### 第二步：审核退款申请

**审核通过（agreeRefundApply）**：

前置校验：
- 申请单必须存在且状态为"待审核"(10)
- 否则提示"申请单不存在或状态不允许审核"

审核通过后：
- `flow_status → 20`（审核通过）
- 设置退款方式（`refundType`）
- 如果是手动退还 → `refund_status` 直接设为30（已退款），不调用支付接口
- 如果是原路退回且 `refund=true` → 立即执行 `doRefund()`
- 发布 `RefundApplyApprovedEvent` 事件（触发后续业务联动）

**审核拒绝（rejectRefundApply）**：

前置校验：
- 申请单必须存在且状态为"待审核"(10)
- 备注不能为空 → "请填写备注"

审核拒绝后：
- `flow_status → 30`（审核不通过）
- 发布 `RefundApplyApprovedEvent` 事件（`agree=false`）

### 第三步：执行原路退款（doRefund）

**前置校验**：
- 审核状态必须是"审核通过"(20)
- 退款状态必须是"未退款"(10) 或 "退款失败"(40)（允许重试）
- 退款方式必须是"原路退回"
- 任一条件不满足 → 提示"申请单状态异常, 无法退款"

**退款执行**：
1. 根据支付渠道选择退款服务：支付宝 `AlipayPaymentService` 或微信 `WxRefundService`
2. 调用退款接口（传递订单号、微信交易号、退款金额、退款单号）
3. 成功 → 设置 `refund_status = 20`（退款中），等待回调
4. 失败 → 捕获异常，调用 `makeApplyRefundFail` 标记退款失败

### 第四步：退款回调处理

微信/支付宝退款成功后，通过 `RefundCompletedEvent` 事件回调：
- `refund_status → 30`（已退款）
- 发布 `RefundApplySucceedRefundedEvent` 事件（触发订单状态更新、收益结算等下游业务）

### 第五步：退款重试与手动退款

**重试退款（retryRefund）**：
- 用于原路退款失败后重新发起
- 直接调用 `doRefund()`，重走校验和退款流程

**手动退款（manualRefund）**：
- 用于原路退款多次失败后，改为线下退款
- 前置条件：审核通过(20) + 退款失败(40)
- 直接设置 `refund_status = 30`（已退款），`refund_type = Manual_Refund`
- 发布 `RefundApplyChangeRefundTypeEvent` 事件（记录退款方式变更）

## 支付渠道解析逻辑

退款时如何确定使用哪个支付渠道（微信还是支付宝）：

1. 优先使用审核请求中传入的 `payChannel`（瞬态字段，不落库）
2. 如果未传入，查询订单的最新收入流水（`OrderFlow.Type=Income`），取其支付渠道
3. 这是为了兼容「跨渠道重复支付」场景：同一订单可能先用微信支付后退款，再用支付宝支付

## 自动退款机制

当 `OrderRefundApplyCreateEvent` 携带 `autoRefund=true` 时：
1. 先创建退款申请
2. 自动构造 `AgreeRefundApply` 请求（审核通过 + 原路退回 + 退款）
3. 自动调用 `agreeRefundApply`
4. 如果自动退款失败 → 记录 error 日志，不中断主流程（管理员需关注）

## 异常处理

| 情况 | 系统处理方式 |
|------|-------------|
| 同一订单重复退款 | 提示"该订单已提交过退款信息" |
| 申请单不存在或状态不允许审核 | 提示"申请单不存在或状态不允许审核, 请联系管理员" |
| 拒绝退款未填备注 | 提示"请填写备注" |
| 申请单状态异常无法退款 | 提示"申请单状态异常, 无法退款" |
| 非原路退回类型调用原路退回 | 提示"当前退款申请单非原路退回申请单, 无法进行原路退回操作" |
| 退款申请单未找到（重试） | 提示"未找到退款申请单, 无法重试退款" |
| 手动退款条件不满足 | 提示"未找到退款申请单或申请单状态异常, 无法退款" |
| 自动退款失败 | 记录error日志，管理员需关注，不中断业务流程 |
| 退款回调收到无效退款单号 | 抛出异常"无效的退款单号" |
| 退款申请单更新冲突（乐观锁） | 提示"退款申请信息已变更，请刷新后重试" |

## 关联功能

- **订单管理**（ykj-service-order）：退款申请以订单号关联
- **支付服务**（微信/支付宝）：原路退款通过支付渠道接口执行
- **事件总线**：退款各阶段发布事件，触发订单状态更新、收益结算、通知发送等
- **管理后台退款审核**：`reimburselist` 页面展示退款审核列表
