# 系统配置历史与回滚流程（服务端）

> 文档版本 1.0 | 面向读者：系统管理员、运维人员

## 概述
系统配置的每次修改都会自动记录历史快照，管理员可以查看配置变更历史，并支持将配置回滚到任意历史版本。回滚操作本身也会产生新的历史记录。

## 触发条件
- 管理员需要追溯某次配置变更的详情
- 新配置导致系统异常，需要回退到之前的版本
- 接口路径：`/mgmt/setting-histories` 或 `/mgmt/setting/histories`

## 步骤详解

### 第一步：配置修改触发历史记录
当管理员通过"全局配置管理"页面修改任意配置项时，系统自动执行以下流程：

1. **校验阶段**：
   - 检查配置项是否存在于配置定义中，不存在的配置项会返回"不存在的配置项：xxx"
   - 校验修改后的值是否符合该配置项的规则（如类型、取值范围等）
   - 如果新旧值相同，跳过该配置项（不产生历史记录）

2. **保存阶段**：
   - 将新值写入配置表
   - 记录变更详情（配置项名称、标题、类型、旧值、新值）
   - 将此次所有变更打包成一条历史记录存入历史表

3. **通知阶段**：
   - 发布 `SettingUpdatedEvent` 事件，通知系统中监听该事件的模块（如缓存刷新等）

### 第二步：查看配置变更历史
调用 `GET /mgmt/setting-histories` 分页查询历史记录，返回字段包括：
- 记录ID
- 变更详情列表（每次可能修改多个配置项）
- 操作人
- 操作时间

需要的操作权限：`sys:setting-histories:read`

### 第三步：回滚配置到历史版本
调用 `POST /mgmt/setting-histories/_restore` 将配置恢复到历史快照中的状态。需要提供：
- **回滚ID**：必填，对应历史记录的主键ID

**回滚流程**：
1. 根据ID查询历史记录，如果不存在返回"回滚历史数据不存在"
2. 取出历史记录中保存的"旧值"，作为新值重新写入
3. 回滚操作同样触发配置更新流程（校验+保存+历史记录+事件发布），即回滚本身也会产生一条新的历史记录

需要的操作权限：`sys:setting-histories:restore`

## 数据模型
- **SysSettingHistoryEntity**：配置历史表
  - `id`：主键
  - `updated_settings`：变更详情（JSON格式，包含每个配置项的名称、标题、类型、旧值、新值）
  - `create_by`：操作人
  - `create_time`：操作时间

## 异常处理

| 情况 | 系统处理方式 |
|------|-------------|
| 回滚ID不存在 | 后端拦截，返回"回滚历史数据不存在" |
| 回滚ID为空 | 前端校验，提示"回滚历史ID不能为空" |
| 配置项不存在 | 后端拦截，返回"不存在的配置项：xxx" |
| 配置值不符合规则 | 后端校验拦截，返回具体规则错误 |
| 新旧值相同 | 静默跳过，不产生历史记录 |

## 关联功能
- **全局配置管理**：管理后台的"系统管理 → 全局配置"页面，修改配置后自动产生历史记录
- **配置校验器（SettingValidator）**：不同类型的配置有不同的校验规则，回滚时同样经过校验
