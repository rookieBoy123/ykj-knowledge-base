# 系统配置批量更新与历史还原流程（服务端）

> 文档版本 2026-07-03 | 面向开发人员和技术支持

## 概述

系统配置管理提供统一的全局参数管理能力。管理后台可以批量修改多个配置项，系统会对每次修改自动生成历史记录（快照），支持将配置回滚到任意历史版本。修改完成后会发布配置变更事件，通知相关子系统刷新缓存。

## 触发条件

- 运营人员在管理后台修改全局配置
- 需要回滚某次错误配置修改
- 系统初始化时加载所有可管理的配置项

## 步骤详解

### 第一步：加载可管理配置项
管理后台请求获取所有配置项列表（`GET /mgmt/settings`）。系统遍历所有已注册的配置定义，返回一个树形结构的配置列表：
- 一级分类（如"问诊设置"、"支付设置"）
- 二级分类（如一級下的具体功能分组）
- 叶子节点为具体的配置项（property + value + type + constraint）

每个配置项包含：属性名（property）、当前值（value）、配置类型（type：Boolean/Text/URL/Select/Decimal/Json/File/TimeRange/DateRange）、显示条件（dependOn）、校验规则（constraint）、可见性（visible）。

### 第二步：提交批量修改
管理后台提交批量更新请求（`POST /mgmt/settings -> batchUpdate`）：

**请求校验**（顺序执行）：
1. **空列表检查**：如果提交的更新列表为空，返回"更新配置项不能为空"
2. **配置项存在性检查**：遍历更新列表中的每个 property，在配置定义表中查找。不存在的 property 会返回"不存在的配置项：xxx"
3. **配置值合法性校验**：对每个更新项调用 SettingValidator 进行类型和格式校验（如URL格式、数值范围、枚举值匹配等）
4. **相同值跳过**：如果新值和旧值完全相同，跳过该项不执行更新

**更新执行**：
1. 将新的 property-value 写入数据库配置表（saveOrUpdate）
2. 对每个真正变更的配置项记录变更前后值
3. 一条修改可能包含多个配置项，对所有变更生成一条统一的历史记录

### 第三步：生成修改历史
修改完成后，系统自动生成一条历史记录（`sys_setting_history` 表），包含：
- **变更明细**：JSON格式存储，包含每个变更项的类型、标题、属性名、旧值、新值
- **操作人**：记录当前登录的用户名
- **操作时间**：精确到秒的时间戳

### 第四步：发布配置变更事件
更新完成后，系统向事件总线发布 `SettingUpdatedEvent` 事件（一次性发布所有变更项的事件）。各订阅模块收到事件后：
- 刷新本地缓存中的配置值
- 更新内存中的全局参数

### 第五步（可选）：查看修改历史
管理后台分页查询历史记录（通过 `SysSettingMgmtService.findPages`）：
- 查询 `sys_setting_history` 表
- 返回字段：ID、变更明细（updatedSettings JSON）、操作人、操作时间
- 按创建时间倒序排列

### 第六步（可选）：还原历史版本
管理员点击「还原」时，系统执行以下流程：
1. 根据历史记录ID查询出对应的历史快照
2. 检查记录是否存在，不存在则返回"回滚历史数据不存在"
3. 从 `updatedSettings` JSON 中提取每个配置项的 property 和 oldValue
4. 将这些值作为新的更新项，递归调用 `updateSettings` 方法（即回滚本质上也是一次"更新"操作，同样会生成新的历史记录）

## 异常处理

| 情况 | 系统处理方式 | 来源 |
|---|---|---|
| 提交空更新列表 | 返回"更新配置项不能为空" | 前端拦截+后端二次校验 |
| 提交不存在的配置项 | 后端日志记录+返回"不存在的配置项：xxx" | 后端 Service |
| 配置值格式不合法 | 返回校验失败的具体错误信息 | SettingValidator |
| 新值与旧值相同 | 静默跳过，不计入变更 | 后端 Service |
| 历史记录不存在 | 返回"回滚历史数据不存在" | 后端 Service |

## 状态流转

```
配置修改提交 → 存在性校验 → 格式校验 → 值比较 → 写数据库 → 写历史记录 → 发布事件 → 各模块刷新缓存
                                    ↓（相同）
                                   跳过
```

## 关联功能

- **系统配置定义**：配置项的元数据（类型、校验规则、依赖关系）由 SettingService 统一管理
- **管理后台-配置管理页面**：前端通过该接口加载和提交配置
- **缓存刷新**：关联 `ykj-service-order`、`ykj-service-inquiry` 等业务模块，收到事件后刷新自身缓存
- **设置历史查询**：`MgmtSysSettingHistoryController` 提供历史分页查询接口
