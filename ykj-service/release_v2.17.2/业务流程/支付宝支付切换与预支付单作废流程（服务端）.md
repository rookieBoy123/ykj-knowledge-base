# 支付宝支付切换与预支付单作废流程（服务端）

> 文档版本 2026-07-05 | 面向运营/技术支持/产品

## 概述

当用户从一种支付方式切换到另一种（如从微信切换到支付宝），系统会自动作废之前的预支付单，防止旧支付渠道的回调干扰当前订单。同时，系统在每次预支付前会检查订单有效期和历史渠道支付状态，确保不会出现重复支付或过期订单被支付的场景。

## 触发条件

- **用户切换支付方式**：用户在收银台从微信支付切换到支付宝（或反过来）后点击"确认支付"
- **小程序倒计时归零**：支付倒计时归零后，小程序主动调用过期接口
- **预支付接口调用**：每次选择支付方式并确认支付时触发

## 核心机制

### 1. 取消预支付单（cancelPayment）

用户在小程序切换支付方式后，旧支付渠道的预支付单会被作废：

- 小程序调用 `POST /api/orders/_cancel-payment` 接口，传入订单编号和要取消的支付渠道
- 服务端取消指定渠道的预支付单（如微信支付生成的二维码失效）
- 如果之前生成了二维码，对方扫码会提示订单不存在或已过期

### 2. 订单有效期拦截

预支付接口新增了过期时间校验：

- 每次预支付前，检查订单的 `expirationTime`
- 如果当前时间已超过过期时间，直接返回 40002 错误码（订单已失效）
- 这条拦截在"订单非待支付状态"检查之前，优先于状态检查
- **目的**：防止定时任务改状态后（订单已被标记为过期），用户的支付回调仍然触发退款

### 3. 历史渠道查单机制

支付前会主动查询各支付渠道的真实状态：

- 调用微信/支付宝的订单查询接口，确认之前是否有支付成功但回调未到达的情况
- 如果查到已支付 → 自动补单（推进订单状态），然后拦截本次支付
- **目的**：防止"用户用微信付了钱但没收到回调，又用支付宝付了一次"的重复支付

### 4. 订单过期主动触发（expire）

新增了客户端主动触发的过期机制：

- 小程序支付倒计时归零时，调用 `POST /api/orders/{orderNo}/expire`
- 服务端校验：已取消/已过期/已退款 → 幂等返回成功（不报错）
- 非待支付状态 → 返回"当前订单状态不允许过期操作"
- 待支付状态 → 标记为超时取消（不触发退款，因为订单从未支付过）

### 5. 重复支付自动退款

支付回调处理时，如果发现订单已完成（已支付状态），说明发生了重复支付：

- 先保存本次的支付流水（保证有收入记录）
- 然后自动发起退款申请，将多付的金额原路退回
- 退款原因标记为"订单重复支付, 系统自动退款"
- 检查退款记录是否已存在（通过 `pay_order_no` ），避免重复退款

### 6. 多流水退款处理

取消订单时，如果订单有多笔收入流水（如分批支付），系统逐条处理：

- 遍历所有收入流水，逐条创建退款申请
- 每条流水独立处理：单条失败不会导致整个取消流程回滚
- 检查流水是否已通过重复支付退款处理过（跳过已退款的流水）
- 最后用乐观锁更新订单状态，防止覆盖并发付款成功的结果

## 状态流转

```
待支付 → [用户支付并回调] → 已支付 / 已完成
待支付 → [倒计时归零/过期] → 已过期（不可恢复）
待支付 → [切换支付方式] → 旧预支付单作废 → 仍为待支付
退款中 → [所有退款被驳回] → 恢复到退款前状态（originFlowStatus）
```

## 异常处理

| 情况 | 系统处理方式 |
|------|------------|
| 预支付时订单已过期 | 返回 40002 错误码，前端弹出"订单已取消"弹窗 |
| 预支付时订单已支付 | 返回 40001 错误码，前端弹出"该订单已支付成功"弹窗 |
| 取消支付但订单不存在 | 后端返回"订单不存在"错误 |
| 重复支付（支付两次） | 系统自动退款多付部分，标记为"订单重复支付, 系统自动退款" |
| 取消订单时乐观锁冲突 | 返回"订单状态已变更，取消失败"，前端提示用户 |
| 多流水退款中某条失败 | 记录错误日志，继续处理下一条，不中断整体流程 |
| 过期接口重复调用 | 幂等处理，已过期状态直接返回成功 |

## 关联功能

- **支付收银台**：小程序 hisPay 页面，用户选择支付方式并确认
- **支付宝教程页**：小程序 aliPay 页面，指导用户保存二维码后用支付宝扫码
- **订单管理**：后台订单列表可查看订单当前状态和流水记录
- **退款审核**：后台退款审核页面可查看由重复支付触发的自动退款记录
- **定时任务**：后台定时扫描过期订单，与客户端主动过期互补
