# CA签章用户生命周期管理流程（服务端）

> 文档版本 2026-07-05 | 面向系统管理员和产品经理

## 概述

CA签章是电子处方的法律效力保障——医生开具的每一张电子处方都需要经过CA（数字证书认证）签章才能生效。CA签章用户管理负责医生在CA平台上的**账号创建、实名认证、签章设置、密码管理、手机号变更、账号注销**等全生命周期操作。

医生在CA平台上有一个唯一的用户编号（UserNo，25位随机字符串），同一人在不同平台（如微信小程序、PC工作台）上共享同一个CA账号，但在每个平台上有独立的OpenId和认证状态。

## 核心概念

| 概念 | 说明 |
|------|------|
| UserNo | 用户的唯一标识，25位Base64编码字符串，全局唯一，由系统随机生成 |
| OpenId | 用户在特定CA平台上的标识，由CA平台返回 |
| Platform | CA平台类型枚举，当前系统支持多个CA供应商（策略模式） |
| AuthStatus | 认证状态：NotAuth（未认证）、Authing（认证中）、Succeed（认证成功）、Failure（认证失败） |
| AppId | 平台应用ID，用于区分同一平台下的不同应用 |

## 步骤详解

### 第一步：创建CA账号

1. 系统管理员或医生本人触发账号创建（通常在医生注册/审核通过时）
2. 系统传入用户姓名、身份证号、手机号等信息
3. 系统首先查询该姓名+身份证号是否已存在：
   - **已存在**：复用已有的UserNo，仅创建该平台的绑定记录
   - **不存在**：生成新的25位UserNo，创建用户实体（姓名、身份证、手机号均**加密存储**）
4. 调用CA平台的同步接口（`CaUserStrategy.sync()`），传入用户信息和UserNo
5. CA平台返回OpenId（平台侧的用户ID）
6. 创建 `CaUserPlatform` 绑定记录：UserNo + OpenId + Platform + AppId，认证状态设为"认证中"
7. 返回CaUser对象给调用方（此时 `enable=false`，需等待认证完成后才能使用）

### 第二步：CA实名认证回调

1. CA平台认证完成后，异步回调系统
2. 系统根据OpenId查找绑定记录（必须匹配 Platform + AppId）
3. 更新认证状态：
   - 认证成功：`authStatus = Succeed`，保存签章图片（stamp）
   - 认证失败：`authStatus = Failure`，保存失败原因（failReason）
4. 发布 `UserAuthEvent` 事件，通知业务系统认证结果

### 第三步：密码设置

1. 医生在前端设置6位CA签章密码
2. 系统首先检查当前平台是否允许用户修改配置（`allowUpdateSetting()`）——不允许会提示"暂不支持用户修改配置"
3. 如果用户还没有签章图片，系统自动生成一个**默认签章图片**（使用用户姓名的文字转图片）
4. 密码使用 **MD5加密**（不可逆）后存储
5. 更新绑定记录中的密码字段

### 第四步：签章图片设置

1. 医生设置手写签章图片
2. 系统先验证密码是否正确——密码错误提示"您的密码验证不通过, 请重新输入"
3. 密码验证通过后，更新签章图片（signImage字段）
4. 签章图片以Base64格式存储

### 第五步：签章验证（签名时调用）

1. 医生在签名处方时，输入签名密码
2. 系统调用 `verifyPassword()` 验证密码——将输入的密码MD5加密后与存储的MD5对比
3. 验证通过则允许签章，不通过则拒绝

### 第六步：手机号变更

1. 医生需要更换绑定的手机号
2. 系统根据UserNo查找当前用户
3. 如果用户不存在，静默跳过（仅记录warn日志，不中断流程）
4. 新手机号**加密存储**到数据库
5. 调用CA平台同步接口，更新用户资料中的手机号
6. 同步成功后，认证状态重置为"认证中"——需要CA平台重新认证
7. 如果CA同步失败，回滚手机号更新并提示"变更手机号码失败，请稍后重试"

### 第七步：账号注销

1. 管理员或医生本人触发账号注销
2. 系统将该用户在平台上的认证状态设为 `NotAuth`（未认证）
3. **不删除**用户数据，仅标记为未认证状态——数据保留用于历史处方追溯

### 第八步：查询用户

**通过UserNo批量查询**：
- 传入多个UserNo，返回对应的CaUser列表
- 需要同时查询 `CaUserEntity`（基础信息）和 `CaUserPlatform`（平台绑定信息），合并返回

**通过UserNo查询认证状态**：
- 传入单个UserNo
- 如果平台绑定记录不存在：返回 `AuthStatus.NotAuth`（兼容资料未同步的场景）
- 如果存在：返回完整的认证状态和签章信息

**通过身份证号查询OpenId**：
- 身份证号加密后查询，用于跨平台用户关联

## 异常处理

| 情况 | 系统处理方式 |
|------|------------|
| CA账号不存在 | 后端拦截，提示"账号不存在" |
| CA平台同步失败（synDto返回null） | 后端记录错误日志，返回null给调用方 |
| CA服务配置异常（无可用策略） | 后端拦截，提示"CA服务配置异常" |
| 平台不允许修改配置 | 后端拦截，提示"暂不支持用户修改配置" |
| 密码验证不通过 | 后端拦截，提示"您的密码验证不通过, 请重新输入" |
| 手机号变更时同步失败 | 后端回滚，提示"变更手机号码失败，请稍后重试" |
| 查询时参数为空 | 后端拦截，提示"参数错误" |
| 用户不存在（变更手机号时） | 静默跳过，仅记录warn日志 |
| 用户不存在（密码设置时） | 静默跳过，仅记录warn日志 |
| 用户认证回调收到无效OpenId | 静默跳过，仅记录warn日志 |
| 系统初始化时存在多平台重复数据 | 自动删除非当前平台的空AppId数据 |

## 安全设计

### 数据加密

| 字段 | 加密方式 | 说明 |
|------|---------|------|
| 身份证号 | `Crypto.encrypt()` | AES加密，数据库不存明文 |
| 手机号 | `Crypto.encrypt()` | AES加密 |
| 密码 | MD5（不可逆） | 不存储明文密码 |

### UserNo生成机制

1. 使用 `SecureRandom` 生成19字节的随机数
2. Base64编码后去掉等号、替换 `/` 和 `+` 为 `_`
3. 截取前25位作为UserNo
4. **递归检查唯一性**：如果生成的UserNo已存在，递归重新生成直到唯一

## 关联功能

- **电子签章**：依赖CA认证状态和签章图片
- **医生审核**：医生审核通过后触发CA账号创建
- **处方签名**：签名时调用密码验证
- **医生更换手机号**：触发手机号变更流程
