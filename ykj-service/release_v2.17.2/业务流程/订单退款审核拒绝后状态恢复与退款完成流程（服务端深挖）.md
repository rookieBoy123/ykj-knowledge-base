# 订单退款审核拒绝后状态恢复与退款完成流程（服务端深挖）

> 文档版本：2026-07-05 | 面向读者：产品经理、运营人员

## 概述

当订单进入退款流程后，管理员可以在管理后台审核退款申请。审核通过→退款到账→订单标记已退款；审核拒绝→订单状态恢复到退款前的原始状态。本文档描述这个完整闭环的后端设计。

## 触发条件

- **前置条件**：订单已产生支付流水，且已提交退款申请（订单进入退款中状态 flowStatus=89）
- **审核入口**：管理后台 → 财务管理 → 退款审核

## 步骤详解

### 第一步：退款申请提交

1. 用户/管理员提交退款申请时，后端执行：
   - 将订单当前状态保存到 `originFlowStatus` 字段（退款前原始状态，如已支付15、待发货20、已完成60等）
   - 将订单状态改为"退款中(89)"
2. 发布退款申请事件，由退款子系统处理

### 第二步：管理员审核

- **审核通过**：退款子系统执行退款操作，退款到账后调用 `completeRefund` 接口
- **审核拒绝**：调用 `restoreOrderFlowStatus` 接口恢复订单状态

### 第三步：审核通过 — 退款完成

1. 调用 `completeRefund(orderNo, refundAmount)` 方法
2. 该接口操作：
   - 订单状态从"退款中(89)"变更为"已退款(90)"
   - 将 `originFlowStatus` 字段清空（已完成退款，不再需要恢复）
   - 通过 LambdaUpdateWrapper 直接更新，不使用乐观锁（避免与退款单更新冲突）

### 第四步：审核拒绝 — 状态恢复

1. 调用 `restoreOrderFlowStatus(orderNo)` 方法
2. 该接口操作：
   - 校验当前订单必须是退款中状态，否则报错"当前订单非退款中状态，无需恢复"
   - 校验 `originFlowStatus` 不为空，否则报错"未找到退款前的订单状态，无法恢复"
   - 将订单状态恢复为退款前的原始状态
   - 将 `originFlowStatus` 字段清空
3. 恢复后，订单回到退款前的状态继续流转

## 异常处理

| 情况 | 系统处理方式 |
|------|-------------|
| 当前订单非退款中状态 | 抛出错误"当前订单非退款中状态，无需恢复" |
| originFlowStatus 为空 | 抛出错误"未找到退款前的订单状态，无法恢复" |
| 恢复操作失败（数据库更新返回0） | 抛出错误"恢复订单状态失败"（内部错误） |

## 关键设计

1. **originFlowStatus 字段**：退款前原始状态的"快照"。退款中期间该字段保存原始状态值，退款完成后清空。这样即使审核过程耗时很长，也不影响状态恢复的正确性
2. **不使用乐观锁**：`restoreOrderFlowStatus` 使用 `LambdaUpdateWrapper` 直接更新，而非乐观锁的 `updateById`。这是因为退款审核是串行操作（同一时间只有一个退款申请在处理），不需要乐观锁防并发
3. **状态清除**：恢复或完成退款后都会清空 `originFlowStatus`，防止字段遗留导致混淆

## 状态流转图

```
正常状态（15/20/30/40/60）
    ↓ 提交退款申请
退款审核中（89）  [originFlowStatus = 原始状态]
    ↓
    ├─ 审核通过 → 已退款（90） [originFlowStatus 清空]
    └─ 审核拒绝 → 恢复原始状态 [originFlowStatus 清空]
```

## 关联功能

- **订单取消退款**：取消订单时也可能触发退款申请，`originFlowStatus` 同样生效
- **退款审核管理**：管理后台审核列表 → 详情查看 → 审核操作
- **订单退款补偿**：退款结果补偿查询时间为 5 分钟（REFUND_RESULT_COMPENSATION_MINUTES）
