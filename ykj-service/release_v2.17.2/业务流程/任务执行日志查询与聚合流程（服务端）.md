# 任务执行日志查询与聚合流程（服务端）

> 文档版本 2026-07-04 | 面向读者：产品/技术 | 服务模块：ykj-service-system

## 概述

系统每次执行定时任务（Cron Job）时，自动记录执行日志，包括任务类型、执行时间、耗时、执行结果等。管理后台可查询这些日志用于监控和排查。接口通过聚合任务定义和日志数据，返回完整的执行记录。

## 接口清单

| 操作 | 接口路径 | 方法 | 权限 |
|------|---------|------|------|
| 分页查询 | `/mgmt/scheduler/logs` | GET | `sys:scheduler:logs` |

## 核心规则

### 1. 日志分页查询

查询参数：
- `jobId`：按任务ID筛选（可选）
- `status`：按执行状态筛选，1=成功、2=失败（可选）
- `curPage`、`pageSize`：分页参数

### 2. 数据聚合

日志表（`sys_scheduled_log`）只存储 `jobId`，不存储任务名称和描述。查询时分两步：

1. **查询日志**：从日志表获取分页数据
2. **聚合任务信息**：提取所有 `jobId`，批量查询任务表（`sys_scheduled_job`），将任务的类型（`jobType`）、名称（`jobName`）、描述（`description`）拼接到日志记录中

这样即使任务已被删除或修改，日志仍能正常显示（任务不存在时对应字段为空）。

### 3. 日志数据结构

| 字段 | 说明 |
|------|------|
| jobType | 任务类型枚举（如 CRON、SIMPLE 等） |
| jobName | 任务名称 |
| description | 任务描述 |
| params | 执行参数（JSON 字符串） |
| scheduleTime | 计划执行时间 |
| fireTime | 实际执行时间 |
| cost | 执行耗时（毫秒） |
| status | 执行状态（1=成功, 2=失败） |
| error | 失败原因（仅失败时有值） |

### 4. 日志记录时机

任务执行完成后，由调度框架自动记录：
- 成功：记录耗时、状态=成功
- 失败：记录耗时、状态=失败、异常堆栈信息

## 异常处理

| 异常情况 | 返回信息 |
|---------|---------|
| 任务ID不存在 | 日志正常显示，但任务类型/名称/描述为空 |
| 任务已被删除 | 同上述情况，历史日志不依赖任务定义 |
| 无权限 | 403 权限拒绝 |

## 状态流转

执行状态为简单的成功/失败二元状态，无复杂流转：
```
任务触发 → 执行中 → 成功（status=1）
                   → 失败（status=2，记录error信息）
```
