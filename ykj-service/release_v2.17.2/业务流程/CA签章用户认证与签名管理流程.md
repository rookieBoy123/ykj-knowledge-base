# CA签章用户认证与签名管理流程

> 文档版本 1.0 | 面向产品/运营人员

## 概述
CA签章是医生开具电子处方的法定电子签名环节。本流程覆盖医生CA账号的创建、资料同步、认证审核和签名使用的全链路后端逻辑。系统支持多CA平台切换（如医网信、源康健PaaS、放心签），不同平台的策略通过统一的账号体系管理。

## 前置条件
- 医生已在系统注册
- CA签章功能已开通（平台配置正确）
- 医生提交了实名认证资料（姓名+身份证号）

## 核心流程

### 1. 创建CA账号
当医生首次需要进行电子签名时，系统自动触发CA账号创建：
- 根据姓名+身份证号（加密存储）查重，同一身份证只生成一个CA账号
- 生成唯一用户编号（25位、Base64编码、基于19字节随机数）
- 调用对应CA平台的同步接口，将医生资料推送到CA厂商
- 记录平台信息（平台类型+AppId+OpenId），区分不同CA厂商

### 2. 认证流程
医生在前端发起认证请求后，后端处理：
- 根据用户编号查询CA账号，校验账号存在性
- 检查该平台下的认证状态
- 调用CA厂商接口获取认证信息（含签章图片）
- 对于资料尚未同步的账号，返回"未认证"状态（兼容旧数据）

### 3. 认证结果处理（异步回调）
CA厂商认证完成后回调系统：
- 更新认证状态：成功/失败
- 失败时记录失败原因
- 更新认证子状态（caSubStatus）
- 成功时保存签章图片（stamp）
- 发布用户认证事件（UserAuthEvent），通知系统内其他模块更新医生CA状态

### 4. 签名使用
医生在开方签名时：
- 系统从CA平台拉取签章图片
- 初始状态为"未启用"（enable=false），需认证通过后激活
- 多平台切换兼容：系统初始化时自动补齐历史数据的AppId字段，清理跨平台重复数据

## 状态流转
```
未同步 → 已同步未认证 → 认证中 → 认证成功 → 可签章
                              → 认证失败（含失败原因）→ 可重新发起
```

## 异常处理
| 情况 | 系统处理方式 | 备注 |
|------|------------|------|
| CA服务配置异常 | 抛系统内部错误"CA服务配置异常" | 检查平台配置 |
| 账号不存在 | 返回"账号不存在" | 需确认是否已创建账号 |
| CA同步失败 | 日志记录+返回null，不阻塞业务 | 同名同身份证不会重复创建 |
| 认证失败 | 记录失败原因，通知前端展示 | 医生可重新发起认证 |
| 跨平台数据冲突 | 初始化时自动清理非当前平台的空AppId数据 | 系统启动时自动执行 |

## 关键设计
- 身份证号全程加密存储，调用`crypto.encrypt()`加密
- CA厂商对接采用策略模式，切换平台只需改配置
- 异步回调处理认证结果，不阻塞用户操作
- 签章图片URL存储在平台记录中

## 关联功能
- 医生处方签名：开具处方时触发签名流程
- 管理后台医生列表：查看CA认证状态、同步资料、去认证操作
- 处方审核：审核时校验签名有效性
