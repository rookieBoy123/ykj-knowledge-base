# 订单取消多流水退款与重复支付检测流程（服务端）

> 文档版本 2026-07-05 | 面向读者：产品、运营、财务

## 概述

v2.17.2 版本对订单取消时的退款处理做了重大升级：从原来的"只处理首笔流水"改为"逐条处理所有收入流水"，同时增加了重复支付退款的检测跳过机制，避免因微信/支付宝回调延迟导致的重复退款。

## 触发条件

- **触发时机**：用户主动取消订单、超时自动取消、管理员取消订单
- **代码位置**：`OrderCommandServiceImpl.cancelOrder()`（`ykj-service-order/.../service/impl/OrderCommandServiceImpl.java`）
- **前置条件**：订单状态必须在可取消状态列表中

## 可取消的订单状态

| 状态 | 能否取消 | 备注 |
|------|---------|------|
| 待支付 | ✅ | 直接取消，无需退款 |
| 已支付 | ✅ | 新增支持，取消时自动申请退款 |
| 待发货/待核销 | ✅ | 取消时自动申请退款 |
| 待收货 | ✅ | 取消时自动申请退款 |
| 待使用（已收货） | ✅ | 取消时自动申请退款 |
| 已完成 | ⚠️ 有限支持 | 仅问诊/预约/咨询预约类型可取消 |
| 退款中/已退款/已过期/已取消 | ❌ | 异常终态，不可再取消 |

## 流程详解

### 步骤一：状态校验

系统首先检查订单当前状态是否在允许取消的范围内。特殊处理：
- **已完成状态**的订单，只有问诊（INQUIRY）、预约挂号（RESERVATION）、咨询预约（COUNSEL_APPOINT）三种类型允许取消
- 超时取消（timeoutCancel=true）时，订单状态标记为"已过期"而非"已取消"

### 步骤二：判断是否需要退款

满足以下全部条件时才生成退款申请：
1. 订单状态不是"待支付"（说明已产生支付流水）
2. 传入的 `refundEnabled` 参数为 true
3. 订单存在收入流水记录

### 步骤三：逐条处理收入流水（核心改进）

旧版本只处理首笔收入流水，当存在多笔支付（如部分退款后再支付、微信支付失败换支付宝等场景）时，其他流水会被遗漏。

新版本改为**逐条循环处理**每笔收入流水：

1. 遍历所有收入流水记录
2. 对每条流水，先检查是否已被"重复支付直接退款"处理过（通过 `t_commonrefund_basic.pay_order_id` 关联查询）
3. 如果已处理过，跳过该条流水（日志记录，不重复退款）
4. 未处理过的流水，逐条发布退款申请事件
5. 单条失败不影响其他条：用 try-catch 包裹，失败时记录错误日志继续处理下一笔

### 步骤四：乐观锁更新状态

最后的订单状态更新使用乐观锁机制：
- 如果 `updateById` 返回 0（说明状态已被并发修改，如支付回调恰好到达），提示"订单状态已变更，取消失败"
- 记录退款前原始状态到 `originFlowStatus` 字段，供后续状态恢复使用

### 步骤五：事件发布

取消成功后发布 `OrderCanceledEvent` 事件，触发下游业务处理（如释放号源、通知医生等）。

## 特殊场景：重复支付退款检测

当微信/支付宝回调延迟导致同一订单收到多次支付通知时：

1. 系统保存所有支付流水记录
2. 在取消退款遍历时，逐条检查是否已存在对应的退款记录
3. 已退款过的流水自动跳过，不重复退款
4. 第一次支付的流水仍正常退款

## 异常处理

| 情况 | 处理方式 |
|------|---------|
| 订单状态不在可取消列表 | 提示"订单取消失败，当前状态不可取消" |
| 已完成状态但订单类型不匹配 | 提示"订单取消失败，当前状态不可取消" |
| 某条流水退款事件发布失败 | 记录错误日志，继续处理下一条（不阻断整体流程） |
| 乐观锁更新失败（并发冲突） | 提示"订单状态已变更，取消失败" |
| 订单无收入流水 | 跳过退款流程，直接取消订单状态 |
