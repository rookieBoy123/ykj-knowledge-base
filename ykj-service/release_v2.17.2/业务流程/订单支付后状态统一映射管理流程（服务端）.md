# 订单支付后状态统一映射管理流程（服务端）

> 文档版本 2026-07-05 | 面向读者：产品、运营

## 概述

不同类型的订单在支付成功后进入的默认状态不同。v2.17.2 版本将原本散落在各处的硬编码状态判断统一收敛到 `OrderConstant.PAYED_STATUS` 映射表中，实现支付后状态的集中管理。

## 状态映射关系

| 订单类型 | 支付后默认状态 | 说明 |
|---------|-------------|------|
| 处方订单（PRESCRIPTION） | 已支付 | 处方需要药师审核后才进入待发货 |
| 问诊订单（INQUIRY） | 已支付 | 问诊需要医生接诊后才开始服务 |
| 会诊订单（CONSULTATION） | 已支付 | 会诊需要多方参与，支付后进入已支付 |
| 预约挂号（RESERVATION） | 待发货/待核销 | 预约挂号支付后直接可核销使用 |
| 服务包（SERVICE_PACKAGE） | 待发货/待核销 | 服务包支付后可直接使用 |
| 咨询预约（COUNSEL_APPOINT） | 已支付 | 咨询预约需等待医生确认 |
| 检查检验（INSPECTION） | 已支付 | 检查需等待医院安排 |

## 触发条件

- **触发时机**：订单支付回调成功时、0元订单创建时
- **代码位置**：`OrderConstant.PAYED_STATUS` 映射表（`ykj-service-order/.../constant/OrderConstant.java`）
- **使用方**：`OrderCommandServiceImpl.createOrder()` 0元订单创建、`handlePaymentResult()` 支付回调

## 流程详解

### 0元订单的创建流程

1. 系统创建订单时，检查订单总金额是否为 0
2. 如果金额为 0，根据 `PAYED_STATUS` 映射表确定初始状态
3. 状态为"已完成"的订单自动设置完成时间，状态为"已支付"的仅设置支付时间
4. 不生成支付流水（0元订单无实际资金流转）

### 支付回调处理流程

1. 第三方支付平台（微信/支付宝）异步回调通知支付成功
2. 系统保存支付流水记录
3. 根据 `PAYED_STATUS` 映射表确定支付后的订单状态
4. 状态为"已完成"的订单发布 `OrderCompletedEvent` 事件，触发后续业务流程（如服务激活、通知发送等）

### 特殊场景：重复支付退款

当同一笔订单被重复支付时（非常见场景，通常由支付回调延迟导致），系统会：

1. **先保存**本次支付对应的收入流水记录（保证账务完整性）
2. **再触发**自动退款流程，将重复支付的金额原路退回
3. 订单状态不受重复支付影响，保留首次支付的正确状态

## 异常处理

| 情况 | 处理方式 |
|------|---------|
| 订单来源类型未在映射表中注册 | 兜底使用"已完成"状态（安全策略：宁可多完成也不错失） |
| 支付回调到达时订单已取消 | 金额原路退回，不修改订单状态（通过 `EXCEPTION_ORDER_STATUS` 判断） |
| 重复支付退款失败 | 记录错误日志，人工介入处理 |

## 关联功能

- **订单创建**（`OrderCommandServiceImpl.createOrder()`）
- **支付回调处理**（`handlePaymentResult()`）
- **订单完成事件**（`OrderCompletedEvent` → 下游模块订阅）
- **自动退款**（`RefundApplyServiceImpl`）
- **订单过期**（`expire` 接口 → `cancelOrder` timeoutCancel）
