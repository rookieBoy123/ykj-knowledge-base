# 退款申请审核流程（服务端）

> 文档版本：2026-07-06 | 面向读者：产品/运营人员

## 概述

当订单需要退款时，系统创建退款申请单，由管理人员审核（同意或拒绝）。审核通过后，根据退款类型（原路退回/手动退款）执行不同的退款流程。支持微信支付和支付宝两种支付渠道的退款。

## 触发条件

- **前置条件**：存在已支付的订单，用户或系统发起退款申请
- **发起方式**：前端触发 → 调用服务端 `createRefundApply` 接口

## 步骤详解

### 第一步：创建退款申请单

系统接收退款申请创建请求，执行以下校验：

1. **防重复校验**：同一笔订单号（orderNo）只能创建一条退款申请。如果同一订单号已存在退款申请单，提示"该订单已提交过退款信息"
2. **支付流水校验**：如果提供了微信支付流水号（wxTransactionId），同一笔支付流水只允许创建一条退款申请，防止重复退款
3. **业务流水号校验**：如果提供了自定义流水号（customNo），作为额外防重条件

创建成功后：
- 退款申请单流程状态设为「待审核」
- 退款状态设为「未退款」
- 自动生成退款订单号

### 第二步：审核退款申请

管理人员审核退款申请单，有两个结果：

**同意退款**：
- 必须处于"待审核"状态的申请单才能审核，否则提示"申请单审核失败, 申请单不存在或状态不允许审核"
- 选择退款类型：
  - **原路退回（Bracktrack）**：如果同时勾选了"立即退款"，系统调用支付渠道（微信/支付宝）执行退款
  - **手动退款（Manual_Refund）**：标记退款状态为"已退款"，不执行自动退款
- 审核通过后发布 `RefundApplyApprovedEvent` 事件，通知相关模块处理后续业务

**拒绝退款**：
- 必须填写拒绝备注，否则提示"请填写备注"
- 流程状态设为「审核不通过」
- 发布拒绝事件通知

### 第三步：执行退款（原路退回）

审核通过且选择"原路退回+立即退款"时：
- 根据支付渠道（微信/支付宝）调用对应的退款接口
- 退款成功后，退款状态更新为"已退款"
- 退款失败时，退款状态更新为"退款失败"，记录失败原因
- 支持退款重试：仅"未退款"或"退款失败"状态的申请单可以重试退款

### 第四步：退款状态同步

**退款成功**：通过 `makeApplyRefundSucceed` 更新申请单的退款状态为"已退款"，记录退款时间和备注。

**退款失败**：通过 `makeApplyRefundFail` 更新申请单的退款状态为"退款失败"，记录失败原因。

### 第五步：事件驱动

退款申请的关键节点通过事件机制通知其他模块：

| 事件 | 触发时机 | 作用 |
|------|---------|------|
| RefundApplyApprovedEvent | 审核完成（同意/拒绝） | 通知订单模块更新订单状态 |
| RefundCompletedEvent | 退款完成 | 通知取消关联的服务 |
| RefundApplyChangeRefundTypeEvent | 变更退款方式 | 变更退款类型 |

## 异常处理

| 情况 | 系统处理方式 |
|------|-------------|
| 同一订单重复提交退款 | 提示"该订单已提交过退款信息" |
| 申请单不存在或状态不允许审核 | 提示"申请单审核失败, 申请单不存在或状态不允许审核, 请联系管理员" |
| 拒绝退款时未填写备注 | 提示"请填写备注" |
| 退款申请单更新失败（乐观锁） | 提示"申请单审核失败, 申请单更新失败, 请联系管理员" |
| 退款失败 | 记录失败原因到退款申请单，可重试 |
| 未找到退款申请单 | 记录警告日志，不中断流程 |

## 关联功能

- **订单管理**：退款与订单生命周期紧密关联
- **微信支付退款**：通过 `WxRefundService` 调用微信退款接口
- **支付宝退款**：通过 `AlipayPaymentService` 调用支付宝退款接口
- **事件通知**：退款审核结果通过事件机制通知药品/服务等模块
