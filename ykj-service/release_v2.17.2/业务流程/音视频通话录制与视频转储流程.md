# 音视频通话录制与视频转储流程

> 文档版本：2026-07-03 第5轮 | 面向读者：产品经理、测试人员

## 概述

音视频通话功能基于腾讯云TRTC（实时音视频）服务实现，覆盖医生与患者之间的在线问诊视频通话、远程会诊多方视频通话。系统在通话结束后自动接收腾讯云回调，保存录制视频信息，并通过异步任务将视频从腾讯云临时存储转储到平台自有OSS存储，确保视频长期可回放。

## 触发条件

- **前置条件**：问诊或会诊已创建，医患双方进入视频通话房间
- **入口**：小程序问诊/会诊界面 → 进入视频通话房间
- **参与方**：医生、患者（1对1）或多方（会诊1对多）

## 步骤详解

### 第一步：通话结束 → 腾讯云回调通知

当视频通话结束后，腾讯云TRTC通过HTTP回调接口通知平台。系统支持两种回调版本：

1. **V1版本回调**（handleCallbackForV1）：
   - 事件类型校验：只处理 `event_type=100` 的录制完成事件，其他类型忽略并记录警告
   - 回调签名验证：使用 `TrtcCallbackSigner.validate()` 校验回调来源合法性
   - 从回调体提取：房间ID、用户ID（从streamParam参数的Base64编码中解析）、视频URL（HTTP→HTTPS强制转换）、结束时间（Unix秒级时间戳×1000转毫秒）、文件格式
   - 将原始回调体（raw）完整存入数据库

2. **V2版本回调**（handleCallbackForV2）：
   - 事件类型校验：只处理 `event_type=311` 的录制完成事件
   - 数据结构不同：从 `eventInfo.payload.body` 中提取用户ID、文件ID、视频URL、结束时间、房间ID
   - V2版本不需要签名验证（TRTC升级后回调格式不同）

两种版本都执行相同的后续处理：将回调信息写入数据库 → 发布领域事件 `RtcVideoCallbackEvent`（用于触达其他业务模块）。

### 第二步：保存视频记录

回调后，系统将录制信息存入数据库：
- 房间ID、用户ID、文件ID
- 视频URL（腾讯云临时地址，有时效性）
- 文件格式（从URL后缀提取）
- 结束时间
- 原始回调体（用于问题排查）

### 第三步：视频资源转储

由于腾讯云的视频URL有时效性（过期后无法访问），系统提供**定时转储任务**将视频从腾讯云转移到自有OSS存储：

1. **分布式锁保护**：使用Redis公平锁（`video:transferred-lock`），确保同一时刻只有一个进程执行转储，防止并发重复处理

2. **批量处理**：每次最多处理100条待转储记录，只拉取重试次数不超过3次的记录

3. **异步转储**：每条记录开异步线程调用OSS的 `transfer` 方法，将视频从远程URL转存到OSS

4. **转储结果标记**：
   - 成功 → 标记为"已转储"（AlreadyTransferred），记录OSS对象ID
   - 失败 → 标记为"转储失败"（TransferredFailure），增加重试次数+1

5. **智能失败重试**：
   - 如果失败原因是因为**文件不存在**（FileNotFoundException），直接将重试次数设为上限值3次，不再重试，避免无意义的重复尝试
   - 其他异常：重试次数+1，下次任务执行时再次尝试（最多3次）

6. **空URL跳过**：如果视频URL为空，直接跳过该记录，不做转储

## 异常处理

| 情况 | 系统处理方式 |
|------|-------------|
| 回调事件类型不匹配 | 记录警告日志"ignore unsupported event type"，不处理该回调 |
| V1回调签名验证失败 | 签名校验不通过，拒绝回调 |
| 视频URL为空 | 跳过该记录，不做转储 |
| 视频源文件已不存在 | 标记失败且直接置重试次数为3（上限），不再重试 |
| 转储失败（非文件不存在） | 重试次数+1，下次任务执行时再次尝试，最多3次 |
| Redis锁被其他进程占用 | 记录警告日志，本次不做转储，等待下次调度 |
| 保存视频时房间不存在 | 抛出异常"保存视频失败，没找到对应房间" |

## 关联功能

- 问诊视频通话（ykj-service-inquiry）：医生问诊时的1对1视频
- 会诊视频通话（ykj-service-consultation）：多方会诊视频
- OSS存储服务（ykj-core-oss）：视频文件的持久化存储
- 调度任务（scheduler）：定时触发视频转储任务
