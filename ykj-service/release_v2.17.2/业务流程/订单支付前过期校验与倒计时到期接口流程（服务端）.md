# 订单支付前过期校验与倒计时到期接口流程（服务端）

> 文档版本 2026-07-05 | 面向读者：产品、运营

## 概述

v2.17.2 版本在订单支付准备阶段（preparePayment）新增了订单过期时间的二次校验，同时新增了前端倒计时归零时主动调用的订单过期接口（`expire`），确保支付倒计时与后端状态严格同步。

## 触发条件

- **过期校验触发**：用户点击支付按钮，调用 `preparePayment` 接口时
- **倒计时到期触发**：收银台页面倒计时归零，前端调用 `POST /api/order/{orderNo}/expire`
- **代码位置**：
  - 过期校验：`OrderCommandServiceImpl.preparePayment()` 第477-480行
  - 过期接口：`ApiOrderController.expire()`（`ykj-service-server/.../controller/api/ApiOrderController.java`）

## 流程详解

### 支付前过期校验流程

当用户在收银台点击支付按钮时：

1. 系统首先检查订单是否为"待支付"状态
2. **新增检查**：即使状态为"待支付"，还会额外检查 `expirationTime`（订单过期时间戳）
3. 如果当前时间已超过 `expirationTime`，直接拒绝支付并返回错误码 40002："订单已失效！"
4. 这个检查的意图是防止以下竞态窗口：定时任务已将订单改为过期，但前端仍在倒计时内发起支付

### 倒计时到期接口流程

当前端收银台倒计时归零时，前端会先检查支付状态，然后调用此接口：

1. 后端查询订单是否存在
2. 如果订单状态已是异常终态（已取消/已过期/已退款/退款中），**幂等返回成功**（不做重复操作）
3. 如果订单不是"待支付"状态，拒绝操作并提示"当前订单状态不允许过期操作"
4. 如果订单是"待支付"状态，构造取消命令：
   - `timeoutCancel=true`（标记为超时取消）
   - `refundEnabled=false`（不收退款——待支付状态无实际支付）
   - 取消备注："订单超时未支付自动取消。"
5. 调用 `cancelOrder` 执行取消

### 前端倒计时归零的完整流程

```
倒计时归零 → 前端先查支付状态（verifyAlreadyPay）
  ├─ 已支付 → 跳转订单详情（用户在其他端完成了支付）
  └─ 未支付 → 调用 expire 接口 → 弹出"订单已取消"提示
```

## 支付宝支付过期错误码体系

| 错误码 | 含义 | 前端处理 |
|--------|------|---------|
| 40001 | 订单已支付（重复支付） | 弹窗告知用户订单已支付，引导返回 |
| 40002 | 订单已失效/已过期 | 弹窗告知订单已取消，引导重新下单 |
| 40003 | 支付失败 | Toast 提示"支付失败，请稍后重试" |

## 异常处理

| 情况 | 处理方式 |
|------|---------|
| 订单不存在 | 提示"订单不存在" |
| 订单已是异常终态 | 幂等返回成功，不做任何操作 |
| 订单已支付或已发货等非待支付状态 | 提示"当前订单状态不允许过期操作" |
| 并发：支付回调恰好在 expire 之前到达 | 前端先查支付状态，支付成功则不调 expire |
| 过期时间设置异常（null） | `expirationTime` 为 null 时跳过过期检查 |

## 关联功能

- **支付准备**（`preparePayment`）：生成支付二维码、拉取支付参数
- **订单取消**（`cancelOrder`）：超时取消复用取消逻辑
- **支付宝支付教程页**（aliPay）：轮询支付状态+eventChannel 通知
- **收银台页面**（hisPay）：倒计时组件、支付渠道选择、支付结果查询
