# 退款申请单跨渠道分渠道退款流程

> 文档版本 2026-07-06 | 面向读者：系统管理员、财务人员

## 概述

退款申请单模块负责管理所有订单退款的全生命周期。它支持微信支付和支付宝支付双渠道退款，通过记录支付渠道实现跨渠道精准退款，并提供了原路退回、手动退款和重试退款三种退款模式。

## 触发条件

- **创建来源**：订单取消时自动触发、管理员主动发起退款、患者申请退款
- **前置条件**：订单已有已支付状态，对应支付流水已入账

## 步骤详解

### 第一步：创建退款申请单

退款申请单的创建遵循**防重原则**——同一笔支付流水只允许创建一条退款申请：

- 系统校验「订单号 + 微信交易流水号 + 自定义单号」三字段组合是否已存在
- 如果已存在，提示「该订单已提交过退款信息」
- 申请单创建后初始状态为「待审核」，退款状态为「未退款」

### 第二步：审核退款申请

管理员审核退款申请时，需要选择退款类型：

- **原路退回（Bracktrack）**：通过支付渠道原路返回资金，适用于微信/支付宝支付
- **手动退款（Manual_Refund）**：管理员线下退款后手动标记，不经过支付系统

审核规则：
- 只有「待审核」状态的申请单可以审核
- 审核通过后，申请单状态变为「审核通过」
- 如果是原路退回且勾选了"立即退款"，系统自动执行退款
- 如果是手动退款，直接标记退款状态为「已退款」

审核拒绝：
- 必须填写拒绝备注，否则提示「请填写备注」
- 申请单状态变为「审核不通过」

### 第三步：分渠道执行原路退款

系统执行退款前做双重验证：

1. **状态校验**：申请单必须是「审核通过」状态 + 退款状态允许重试（未退款 / 退款失败）
2. **类型校验**：必须是「原路退回」类型，否则提示「当前退款申请单非原路退回申请单，无法进行原路退回操作」

验证通过后，系统根据支付渠道分派退款请求：

- **支付宝渠道**：调用 `AlipayPaymentService.refund()` 发起支付宝退款
- **微信渠道**：调用 `WxRefundService.refund()` 发起微信退款

退款发起后，申请单退款状态标记为「退款中」。

### 第四步：退款回调处理

支付平台异步通知退款结果后，系统处理回调：

- 优先使用回调事件携带的支付渠道（支持跨渠道重复支付场景）
- 如果未携带，回退到查询最新收入流水获取渠道（兼容历史数据）
- 退款成功：标记退款状态为「已退款」，发布 `RefundApplySucceedRefundedEvent` 事件通知下游
- 退款失败：标记退款状态为「退款失败」，记录失败原因

### 第五步：手动退款（原路退回失败后的兜底）

当原路退回失败后（微信/支付宝返回退款失败），管理员可以执行**手动退款**：

- 前置条件：申请单必须同时满足「审核通过」+「退款状态为退款失败」
- 不符合条件时提示「未找到退款申请单或申请单状态异常，无法退款」
- 手动退款直接标记退款状态为「已退款」，不经过支付渠道
- 发布 `RefundApplyChangeRefundTypeEvent` 事件记录退款类型变更

### 第六步：重试退款

退款失败后，管理员可以点击「重试退款」：

- 系统查找对应的退款申请单
- 如果未找到，提示「未找到退款申请单，无法重试退款」
- 重新执行 `doRefund` 流程（走完整的校验 + 分渠道退款）

## 异常处理

| 情况 | 系统处理方式 |
|------|-------------|
| 同一支付流水重复提交退款 | 提示「该订单已提交过退款信息」 |
| 审核时申请单已不是待审核状态 | 提示「申请单不存在或状态不允许审核」 |
| 拒绝时未填写备注 | 提示「请填写备注」 |
| 原路退款时申请单状态异常 | 提示「申请单状态异常，无法退款」 |
| 非原路退回申请单执行原路退款 | 提示「当前退款申请单非原路退回申请单」 |
| 手动退款时申请单不满足条件 | 提示「申请单状态异常，无法退款」 |
| 退款执行异常 | 记录失败原因，自动标记退款失败 |
| 退款回调时退款单号无效 | 提示「无效的退款单号」 |
| 退款申请信息并发变更 | 提示「退款申请信息已变更，请刷新后重试」 |

## 关联功能

- **订单管理**：退款申请单关联订单号，订单取消时自动触发退款
- **交易流水**：退款申请单关联收入流水，通过微信交易流水号追溯
- **支付渠道管理**：微信支付（`WxRefundService`）和支付宝支付（`AlipayPaymentService`）独立退款
- **事件机制**：退款审核通过/退款完成等关键节点发布事件，驱动下游业务（如收益结算、通知发送）
