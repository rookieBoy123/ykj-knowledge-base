# 订单完成事件与跨服务状态联动流程（服务端深挖）

> 文档版本：2026-07-05 | 面向读者：产品经理

## 概述

当订单被标记为"已完成"时，系统发布 `OrderCompletedEvent` 事件，各业务子系统监听到该事件后执行各自的后续操作（如生成收益、更新统计数据等）。本文档描述订单完成事件的设计和使用场景。

## 触发条件

- **触发时机**：订单状态变更为"已完成(60)"时
- **触发方式**：调用 `completeOrder(orderNo)` 方法

## 步骤详解

### 第一步：主动完成订单

1. `completeOrder` 方法校验订单当前状态必须属于以下之一：
   - 已付款(15) - PAYED
   - 待发货(20) - PENDING_SHIPMENT
   - 已签收(40) - RECEIVED
2. 不在以上状态则拒绝："【xxx】状态的订单不允许主动完成，请联系管理员"

### 第二步：状态变更

1. 订单状态设为"已完成(60)"
2. 记录完成时间
3. 通过乐观锁更新数据库

### 第三步：发布完成事件

1. 构建 `OrderCompletedEvent` 对象，包含：
   - orderNo（订单编号）
   - sourceType（订单来源类型：问诊/处方/会诊/预约/服务包/咨询/检验检查）
   - businessNo（业务编号）
   - memberId（会员ID）
   - totalAmount（订单总金额）
2. 通过事件发布器发布，各子系统异步处理

### 第四步：其他触发场景

1. **确认收货（confirmReceipt）**：患者确认收货时，订单直接从已发货改为已完成，同时设置 completeTime
2. **0元订单创建时**：某些0元订单类型在创建时直接设为已完成
3. **定时自动收货**：发货超15天的处方订单自动确认收货并完成

## 事件消费者（各子系统响应）

| 子系统 | 处理动作 |
|--------|---------|
| 收益模块 | 订单完成后计算医生/推介人收益 |
| 通知模块 | 向患者/医生发送订单完成通知 |
| 统计模块 | 更新订单统计数据 |

## 0元订单状态规则

0元订单不再统一标记为"已完成"，而是根据订单来源类型分派不同状态：

| 订单来源类型 | 0元订单状态 |
|-------------|-----------|
| 处方(PRESCRIPTION) | 已付款(15) |
| 问诊(INQUIRY) | 已付款(15) |
| 会诊(CONSULTATION) | 已付款(15) |
| 咨询(CounsEL_APPOINT) | 已付款(15) |
| 检验检查(INSPECTION) | 已付款(15) |
| 预约(RESERVATION) | 待发货(20) |
| 服务包(SERVICE_PACKAGE) | 待发货(20) |
