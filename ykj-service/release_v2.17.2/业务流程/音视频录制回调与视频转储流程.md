# 音视频录制回调与视频转储流程

> 文档版本 2026-07-03 | 面向系统管理员/开发运维

## 概述

当医生和患者通过平台进行音视频通话时，腾讯TRTC（实时音视频）服务会自动录制通话内容。录制完成后，TRTC通过回调通知平台，平台接收回调数据后将视频资源从第三方存储转储到自有OSS存储，确保视频数据的安全和持久保存。

## 触发条件

- **前置条件**：问诊/会诊中发起了音视频通话，且开启了云端录制
- **触发方式**：TRTC录制完成后主动回调平台接口

## 步骤详解

### 第一步：TRTC云端录制

当医生和患者进行音视频通话时，TRTC的云端录制服务开始工作。录制完成后，TRTC生成视频文件并通过回调接口通知平台。

平台支持两种回调协议版本：
- **V1版本**：事件类型码 100，适用于旧版TRTC录制回调
- **V2版本**：事件类型码 311，适用于新版TRTC录制回调

### 第二步：接收回调

平台接收到TRTC回调后，根据事件类型分发处理：

**V1版本处理**：
1. 校验事件类型是否为100（仅处理录制事件，其他事件类型忽略并记录日志）
2. 对回调数据进行签名验证（防止伪造回调）
3. 解析回调参数中的 `streamParam`（流参数），提取用户ID（Base64编码）、房间ID、群组ID
4. 将视频URL中的 `http` 替换为 `https`（安全传输）

**V2版本处理**：
1. 校验事件类型是否为311
2. 直接从事件体中提取房间ID、用户ID、视频URL、录制结束时间、文件格式

### 第三步：存储回调记录

无论V1还是V2，处理完成后都会：
1. 将回调原始数据和解析后的结构化数据存入 `t_trtc_callback` 表
2. 发布 `RtcVideoCallbackEvent` 事件，供其他模块监听处理（如关联到具体问诊记录）

### 第四步：视频资源转储

视频转储是一个独立的后台定时任务：

1. **获取分布式锁**：使用Redis公平锁（`video:transferred-lock`）确保同一时间只有一个进程执行转储
2. **查询待转储视频**：从 `t_rtc_video` 表中查找需要转储的视频记录，限制条件是：
   - 每次最多处理100条
   - 重试次数不超过3次
3. **执行转储**：对每条视频异步执行转储操作：
   - 通过OSS服务将视频从原始URL传输到平台自有存储
   - 转储成功后：更新视频的OSS对象ID，标记为"已转储"
   - 转储失败后：
     - 如果源文件不存在（`FileNotFoundException`）：直接将重试次数设为上限值3，不再重试
     - 其他异常：重试次数+1，下次任务继续尝试（最多3次）
4. **释放锁**

### 第五步：视频保存

除了定时转储，系统还支持主动保存视频：

- **接口入口**：`saveVideo(roomId, userId, videoUrl)`
- **校验逻辑**：根据roomId查找RTC房间，如果房间不存在，返回错误"保存视频失败, 没找到对应房间"
- **保存操作**：在 `t_rtc_video` 表中创建视频记录

### 第六步：管理后台查询

管理员可以通过管理后台查询指定业务编号下的所有视频记录：
- **接口**：`GET /mgmt/rtc/videos?businessNo=xxx`
- **权限**：需要 `rtc:videos:read` 权限

## 状态流转

```
TRTC录制完成 → 回调通知平台 → 存储回调记录 → 发布事件 → 
定时任务拉取待转储视频 → 执行转储 → 成功(已转储)/失败(重试最多3次)
```

## 异常处理

| 情况 | 系统处理方式 |
|:---|:---|
| 回调事件类型不匹配 | 忽略该回调，记录警告日志（V1: 非100，V2: 非311） |
| 转储时源文件不存在 | 直接标记为失败且不再重试（重试次数=3），避免无效重试 |
| 转储时出现其他异常 | 重试次数+1，下次定时任务继续处理，最多重试3次 |
| Redis锁被占用 | 记录日志"视频资源转储失败，redis锁被其他进程占用"，本次跳过 |
| 视频URL为空 | 记录日志"需要转储的资源URL为空"，跳过该条记录 |
| 找不到对应房间 | 返回错误"保存视频失败, 没找到对应房间"，前端提示 |
| V1回调中streamParam为空 | 无法提取userId/roomId，仅记录不完整的回调数据 |
| 视频URL是http协议 | V1处理时会自动替换为https（安全加固）|

## 关联功能

- **问诊/会诊音视频通话**：视频录制发生在医生和患者的实时通话中，通过roomId关联
- **OSS存储服务**：视频最终存储到平台自有OSS，确保数据安全可控
- **RTC房间管理**：视频记录关联到具体的RTC房间（t_rtc_room表）
- **事件发布**：回调处理后发布领域事件，其他模块可监听做进一步处理（如关联问诊记录、生成回放链接）

## 数据模型

| 表 | 关键字段 | 说明 |
|:---|:---|:---|
| t_rtc_video | roomId, rtcUserId, url, videoId, transferred, retryTimes | 视频记录主表 |
| t_trtc_callback | roomId, userId, videoUrl, raw, fileId, fileFormat | 回调原始记录 |

**转储状态枚举（transferred）**：
- `UnTransferred`：待转储（初始状态）
- `AlreadyTransferred`：已转储（成功）
- `TransferredFailure`：转储失败
