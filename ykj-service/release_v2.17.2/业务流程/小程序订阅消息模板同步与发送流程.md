# 小程序订阅消息模板同步与发送流程

> 文档版本 v1.0 | 面向读者：产品与运营人员  
> 服务模块：ykj-service-subscribe

## 概述

平台通过微信小程序订阅消息能力，向患者发送问诊状态变更、处方审核结果、支付提醒等服务通知。本流程涵盖消息模板的自动同步管理和消息的异步发送机制。

## 核心流程

### 一、模板同步

系统通过定时任务或手动触发，将配置的订阅消息模板与微信后台保持同步：

1. **加锁防并发**：使用Redis分布式锁（`subtmpls:sync-lock`），确保同一小程序的同步任务不会被重复执行
2. **类目资质校验**：检查小程序是否具备对应服务类目权限，若未添加则提示"小程序未添加XX，请登录小程序管理后台添加！"
3. **读取模板配置**：从YAML配置文件中读取模板元数据（`classpath:subscribe/schema/XX-hospital.yaml`），包含模板ID、标题、关键字等信息
4. **三方数据对比**：将本地数据库（DB）、微信后台API和YAML配置三方的模板数据进行对比
5. **自动同步**：
   - DB有+API无 → 记录错误日志，不可删除DB记录（否则已订阅用户无法收到消息）
   - DB有+API有+不一致 → 更新DB为API最新数据
   - DB无+API有 → 直接插入DB
   - DB无+API无 → 调用微信API新增模板到微信后台
6. **刷新Redis缓存**：同步完成后清空旧缓存，重新加载模板数据到Redis

### 二、消息发送

当业务事件触发时（如问诊状态变更），系统异步发送订阅消息：

1. 根据消息类型查找对应的模板配置（先从Redis缓存取，缓存未命中从DB取）
2. 组装消息数据：将业务数据按模板关键字顺序填充，通过 `@Keyword(index=N)` 注解标注字段顺序
3. 调用微信API发送订阅消息（`wxService.getSubscribeService().sendSubscribeMsg`）
4. 记录发送日志（成功/失败），包含appId、openId、模板ID、消息体和结果
5. 发送失败时仅记录警告日志，不阻塞业务流程

### 三、发送日志清理

定时任务每隔3个月清理一次发送日志（`clearSendLog`），删除3个月前的日志记录，控制日志表数据量。

## 异常处理

| 情况 | 处理方式 |
|------|---------|
| 小程序未添加对应服务类目 | 后端抛出异常："小程序未添加XX，请登录小程序管理后台添加！" |
| 微信后台模板被删除 | 记录错误日志："未在微信后台配置订阅消息模板，或消息模板已被删除"，不删除DB记录 |
| 模板配置未找到 | 记录错误日志："未配置模板消息元数据"，跳过同步 |
| 模板类目ID不匹配 | 记录错误日志："消息模板类目配置异常"，跳过同步 |
| 新增模板时微信已存在 | 从错误响应中提取priTmplId，直接保存到DB |
| 发送消息时模板不存在 | 记录错误日志："小程序消息模板不存在" |
| 发送消息失败 | 记录警告日志和错误详情，不抛异常 |
| 同步锁被占用 | 记录错误日志："订阅消息模板同步失败，同步锁被其他进程占用" |

## 状态流转

```
YAML配置（模板元数据）
    ↓ 读取
内存（schemaMap按appId缓存）
    ↓ 同步
微信后台（微信管理的订阅模板）
    ↓ 同步
数据库（SubscribeRepositoryPO）
    ↓ 缓存
Redis（subtmpls:{appId} Hash结构）
    ↓ 读取
消息发送（组装消息+调用微信API）
    ↓ 记录
发送日志（SubscribeLogPO，3个月后定时清理）
```

## 关键设计

- **分布式锁**：使用Redisson公平锁，防止多节点并发同步
- **Redis缓存**：模板数据缓存在Redis Hash结构中（key=subtmpls:{appId}），提高读取速度
- **异步发送**：消息发送使用 @Async 异步执行，不阻塞主业务
- **配置驱动**：模板元数据（模板ID、标题、关键字）集中管理在YAML文件中，按类目（如hospital）分类
- **关键字匹配**：通过 @Keyword(index) 注解和反射机制自动匹配业务字段与模板占位符（{{xxx.DATA}}）
- **不可删除原则**：DB中的模板记录不可删除，否则已订阅用户将无法收到消息
