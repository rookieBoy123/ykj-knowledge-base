# 订单过期与倒计时取消流程（服务端深挖）

> 文档版本：2026-07-05 | 面向读者：产品经理、运营人员

## 概述

订单创建后有一个支付倒计时，用户需要在规定时间内完成支付。倒计时归零后，小程序前端会主动调用订单过期接口，后端将订单标记为失效。本文档描述后端订单过期的完整处理逻辑。

## 触发条件

- **前置条件**：订单处于待支付状态（flowStatus=10）
- **触发方式**：小程序前端倒计时归零时，调用 `POST /api/orders/{orderNo}/expire` 接口

## 步骤详解

### 第一步：前端倒计时归零

1. 收银台页面展示剩余支付时间（基于订单的 expirationTime 计算）
2. 倒计时归零后触发 `zltimeup` 方法
3. **并非直接调用过期接口**，而是先通过 `verifyAlreadyPay` 查询订单支付状态

### 第二步：倒计时前的二次确认

1. 如果订单已被支付（如在支付宝 App 中完成支付但回调未到达小程序），走支付成功流程
2. 如果订单未支付，前端调用 `POST /api/orders/{orderNo}/expire` 接口

### 第三步：后端过期接口处理

1. 后端查询订单，不存在则返回"订单不存在"
2. **幂等处理**：如果订单已经是已取消/已过期/已退款等异常终态，直接返回成功（不做重复操作）
3. 如果订单状态不是"待付款"，返回错误"当前订单状态不允许过期操作"
4. 状态合规则执行取消订单命令：
   - cancelRemark = "订单超时未支付自动取消。"
   - timeoutCancel = true（标记为超时取消）
   - refundEnabled = false（已付款的订单才涉及退款，待付款订单无需退款）

### 第四步：订单状态变更

1. 订单状态从"待付款(10)"变更为"已失效(80)"
2. 记录取消失败原因
3. 乐观锁保障并发安全
4. 发布 `OrderCanceledEvent` 事件通知各子系统

## 异常处理

| 情况 | 后端处理方式 |
|------|-------------|
| 订单不存在 | 返回错误"订单不存在" |
| 订单已是异常终态（已取消/已过期/已退款） | 幂等返回成功（不做重复操作） |
| 订单不在待支付状态 | 返回错误"当前订单状态不允许过期操作" |
| 乐观锁冲突（并发操作） | 返回错误"订单状态已变更，取消失败" |
| 前端倒计时归零但订单已支付 | 前端先查支付状态，已支付走成功流程 |

## 与前端交互的精要设计

1. **前端先确认后后端执行**：倒计时归零时前端先查支付状态，避免"用户在最后一秒完成支付但后端标记已失效"的竞态问题
2. **后端幂等保护**：已失效的订单再次调用过期接口直接返回成功，防止前端重试导致报错
3. **状态校验**：只有待付款状态才能执行过期，已支付/已发货等状态拒绝操作

## 关键配置

| 配置项 | 说明 |
|--------|------|
| DEFAULT_ORDER_INVALID_HOUR = 24 | 默认订单有效期 24 小时（若无业务方自定义过期时间） |
| 业务方自定义 expirationTime | 优先使用业务方设置的过期时间，覆盖默认 24 小时 |

## 关联功能

- **支付准备（advance）**：支付前校验订单有效期，过期则返回 40002 错误码
- **订单取消（cancelOrder）**：过期接口内部复用取消订单命令
- **订单退款**：已付款的订单过期触发自动退款，待付款订单过期仅标记失效
