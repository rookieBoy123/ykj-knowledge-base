# 订单支付前校验与重复支付防护流程（服务端深挖）

> 文档版本 1.0 | 面向读者：产品、运营、测试

## 概述

本流程描述订单支付请求发起前系统执行的一系列安全校验，包括订单状态检查、有效期校验、历史支付查询、支付渠道验证和分布式锁机制，确保每笔支付的安全性和唯一性。

## 触发条件

- 用户在小程序或管理后台发起订单支付
- 系统收到第三方支付回调（微信/支付宝/医保）

## 步骤详解

### 第一步：准备支付（advance / preparePayment）

系统收到支付请求后，首先进行三重状态校验：

1. **检查是否已支付**：如果订单状态为"已支付""已完成""已退款"等，直接拦截并返回"当前订单已支付"提示（错误码40001）
2. **检查订单有效性**：只有"待支付"状态的订单才能进行支付，其他状态返回"订单已失效"提示（错误码40002）
3. **检查过期时间**：如果订单设置了过期时间且当前时间已超过，同样返回"订单已失效"

### 第二步：历史支付渠道查单

在正式发起支付前，系统会查询该订单的历史预支付单记录，按渠道（微信/支付宝）逐个向第三方查询真实支付状态：

1. **查询目标**：只查 callbackFlag=0（等待回调）的旧预支付单
2. **查单失败处理**：如果网络抖动或API异常导致查单失败，系统以"体验优先"原则放行，不阻断本次支付
3. **发现已支付处理**：如果查到第三方已支付但系统未入账，系统会在独立事务中同步补单（推进订单状态到已支付），然后拦截本次支付

### 第三步：构建支付渠道列表

系统根据订单类型和系统配置，动态构建可用的支付渠道：

1. **微信支付**：默认支持，默认选中（JSAPI方式），另有微信扫码可选
2. **支付宝**：通过 `paymentProperties.alipay.enabled` 配置开关控制是否显示
3. **医保支付**：仅处方订单支持医保支付，且需要医保支付开关开启，同时通过 `ValidatePayMethod` 校验患者医保信息是否可用

### 第四步：预支付加锁（prepay）

正式发起预支付时，系统通过分布式锁（Redisson公平锁）防止并发支付：

1. **预支付锁**：以 `lock:prepay:{订单号}` 为key，等待5秒，锁定10秒
2. **锁获取失败**：返回"支付处理中，请稍后重试"（错误码40003）
3. **前置清理**：获取锁成功后，先关闭旧的预支付单（含调用第三方关单接口），清除缓存中的支付二维码
4. **渠道校验**：校验用户选择的支付渠道是否在订单支持的渠道列表中，不在则提示"该订单不支持当前支付方式"

### 第五步：支付回调入账（completePayment）

第三方支付成功后回调系统，进行入账操作：

1. **入账锁**：以 `order:complete:{订单号}` 为key，防止回调与主动查询并发
2. **重复支付防护**：如果订单已不是待支付状态（说明之前已入账），检查是否为同一笔支付（通过transactionId/payOrderNo幂等判断）：
   - **同一笔支付**：幂等跳过，不处理
   - **不同支付**：先保存收入流水，然后自动触发退款，将多余款项原路退回
3. **入账处理**：记录支付流水 → 更新订单状态 → 已付款（或已发货/已完成）→ 处方订单自动生成取货码 → 发布订单已支付事件

## 异常处理

| 情况 | 系统处理方式 |
|------|-------------|
| 订单已支付/已完成 | 拦截支付，提示"当前订单已支付"（错误码40001） |
| 订单状态非待支付 | 拦截支付，提示"订单已失效"（错误码40002） |
| 订单已过期 | 拦截支付，提示"订单已失效" |
| 分布式锁获取失败 | 拦截支付，提示"支付处理中，请稍后重试"（错误码40003） |
| 支付渠道不支持 | 拦截支付，提示"该订单不支持当前支付方式" |
| 历史已支付但未入账 | 同步补单后拦截本次支付 |
| 重复支付（不同交易单） | 自动退款，多余款项原路退回 |
| 查单网络异常 | 放行不阻断，保证正常支付体验 |

## 关联功能

- 订单取消 → 自动触发退款申请单生成
- 退款事件 → OrderRefundApplyCreateEvent 异步处理
- 订单已支付事件 → OrderPaidEvent 触发后续流程（发货通知、IM消息等）
- 15天自动收货 → 定时任务 automaticReceiptFor15Days
