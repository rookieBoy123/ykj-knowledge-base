# 订单过期与取消支付接口流程（服务端）

> 文档版本 2026-07-05 | 面向产品/运营/开发

## 概述

本次新增了两个独立的订单操作接口：**订单过期**和**取消支付（作废二维码）**。订单过期接口由小程序倒计时归零时主动调用，取消支付接口允许用户在支付页面主动关闭二维码。两个接口都将原本在前端/定时任务中的逻辑规范化到了服务端。

## 一、订单过期接口

### 触发条件

- 小程序支付倒计时（通常15分钟）归零时自动调用
- 仅「待付款」状态的订单可执行过期

### 步骤详解

1. **小程序倒计时归零**：用户停留在支付页面上，倒计时到0
2. **前端调用过期接口**：`POST /api/order/{orderNo}/expire`
3. **服务端校验**：
   - 订单是否存在 → 不存在则返回「订单不存在」
   - 订单是否已处于异常终态（已取消/已过期/已退款）→ 幂等返回成功（不做任何操作）
   - 订单是否待付款 → 非待付款状态拒绝过期
4. **执行取消**：设置取消原因为「订单超时未支付自动取消」，标记为超时取消，不触发退款
5. **订单状态变为已过期**

### 幂等保护

如果用户网络延迟导致前端重复调用过期接口，已过期的订单在第二次调用时会命中「已处于异常终态」的校验，直接返回成功不报错。

## 二、取消支付接口

### 触发条件

- 用户在支付页面（展示二维码时）点击「取消支付」或关闭支付弹窗
- 任何待付款状态下的订单均可取消支付

### 步骤详解

1. **用户点击取消**：在收银台页面选择取消支付
2. **前端调用接口**：`POST /api/order/_cancel-payment`，传入订单编号和支付渠道（微信/支付宝）
3. **后端作废预支付单**：`cancelPayment` 方法会：
   - 查询该订单对应的预支付记录
   - 调用支付渠道的关闭接口（微信关闭订单 / 支付宝关闭交易）
   - 将预支付单标记为已关闭
4. **前端更新状态**：前端收到成功后刷新订单状态，用户可以重新选择支付方式

### 与订单取消的区别

| 对比项 | 取消支付 | 取消订单 |
|--------|---------|---------|
| 操作对象 | 预支付单（二维码） | 整笔订单 |
| 订单状态变化 | 不变，仍为待付款 | 变为已取消 |
| 是否可以重新支付 | 可以 | 不可以 |
| 是否允许退款 | 不涉及 | 已付金额需退款 |

## 支付状态校验优化

本次还优化了问诊订单支付状态的轮询校验逻辑：

1. **优先查本地状态**：先查订单在数据库中的 `flowStatus`
2. **本地未支付时查第三方**：如果订单有指定支付方式，主动调用微信/支付宝查询接口确认
3. **处理回调丢失**：如果第三方回调因网络问题丢失，轮询查询可以作为兜底，避免订单一直显示待付款

## 异常处理

| 情况 | 系统处理方式 |
|------|-------------|
| 过期的订单已被取消/退款 | 幂等返回成功，不做任何操作 |
| 已支付的订单调用过期 | 返回「当前订单状态不允许过期操作」 |
| 取消支付时支付渠道已关闭 | 返回成功（幂等处理） |
| 订单不存在 | 返回「订单不存在」 |
