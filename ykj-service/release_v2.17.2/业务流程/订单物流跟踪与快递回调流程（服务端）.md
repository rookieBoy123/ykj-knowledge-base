# 订单物流跟踪与快递回调流程（服务端）

> 文档版本：2026-07-04 | 面向读者：产品经理、后端开发、运营人员
> 关联模块：`ykj-service-order`

## 概述

药品订单发货后，系统与快递100物流平台对接，实时获取物流轨迹信息。医生和患者两端均可查询物流详情，且系统对数据做了严格的归属校验——医生只能看自己订单的物流，患者只能看自己订单的物流。

## 参与角色

| 角色 | 操作 | 限制 |
|------|------|------|
| 患者 | 查询自己订单的物流详情 | 只能查看自己会员ID下的订单 |
| 医生 | 查询自己开具订单的物流详情 | 只能查看自己医生ID下的订单 |
| 快递100 | 推送物流状态更新回调 | 通过签名验证才接受 |

## 流程详解

### 1. 物流信息订阅

当药房发货后，系统向快递100平台订阅该快递单号的物流推送。快递100会在物流状态发生变化时主动回调通知系统。

### 2. 快递回调处理（OrderLogisticsController）

快递100通过两个接口回调系统：

**接口一：`/callback/logistics/kd100`**（标准回调）
- 快递100携带 `sign`（签名）和 `param`（加密参数）回调
- 系统通过 `LogisticsService.handleCallback()` 处理：
  1. 验证签名的合法性
  2. 解密 param 中的物流轨迹数据
  3. 更新数据库中对应快递单号的物流详情

**接口二：`/callback/logistics/paas`**（PaaS平台回调）
- 区别于标准回调，PaaS回调直接传输JSON格式数据体
- 签名参数为 null，系统仅校验数据体的合法性

### 3. 医生端物流查询（DoctorLogisticsController）

- 接口：`GET /api/doctor/logistics`
- 参数：`deliverySn`（快递单号）
- **安全校验**：系统先根据快递单号查出所有关联订单，然后逐一比对当前登录医生ID是否与订单中的医生ID匹配。如不匹配，返回"参数错误"（不暴露是否存在的细节）

### 4. 患者端物流查询（PatientLogisticsController）

- 接口：`GET /api/patient/logistics`
- 参数：`deliverySn`（快递单号）
- **安全校验**：系统根据快递单号查出所有关联订单，比对当前登录会员ID是否与订单中的会员ID匹配。如不匹配，返回"参数错误"

### 5. 物流数据存储

物流信息存储在 `OrderLogistics` 表中：
- `nu`：快递单号
- `details`：物流轨迹详情（JSON格式，包含时间、地点、状态描述）
- 查询时系统将JSON字段反序列化为 `OrderLogisticsDetails` 对象列表返回

## 状态流转

```
药房发货 → 系统订阅快递100 → 快递100推送物流更新
                                  ↓
                       系统更新 OrderLogistics 表
                                  ↓
                   医生端/患者端 查询物流详情
```

## 安全设计

| 设计要点 | 说明 |
|---------|------|
| 归属校验 | 医生只能查询自己订单的物流，患者只能查询自己订单的物流 |
| 信息隐藏 | 无权限时统一返回"参数错误"，不暴露订单是否存在 |
| 签名验证 | 快递100回调必须通过签名验证才接受数据 |
| 双回调通道 | kd100（标准签名版）和paas（PaaS平台版）两种回调方式 |

## 异常处理

| 情况 | 系统处理方式 |
|------|-------------|
| 医生查询非自己订单的物流 | 后端拦截，返回"参数错误" |
| 患者查询非自己订单的物流 | 后端拦截，返回"参数错误" |
| 快递单号无物流记录 | 返回 null |
| 快递100回调签名验证失败 | 拒绝更新，不处理回调数据 |
| 快递单号为空 | 直接返回 null，不查询数据库 |

## 关联功能

- **订单创建与支付管理**：订单发货后触发物流订阅
- **药房发货管理**：药房确认发货 → 触发物流信息录入
- **快递100平台**：外部物流轨迹数据源
